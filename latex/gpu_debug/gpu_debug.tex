\noindent{\Large\bfseries\color{nvidiagreen} NVIDIA GPU 调试指南}

\vspace{0.8em}

本文档提供 GPU 错误调试与诊断指南，旨在帮助系统管理员、开发人员和现场应用工程师（FAE）尽快恢复服务器的正常运行。

本文档提供了一套流程框架及相关细节，说明如何开始调试 GPU 服务器上的常见问题。其目的是涵盖数据中心 GPU 运维中最常见的问题，但并非面面俱到。以下信息是多份不同文档和 GPU 系统问题调试最佳实践的综合概要。

本调试流程旨在提供通用指导，可能与您的系统供应商的特定故障排查指南不完全一致。请尽早（而非等到问题恶化后）与您的系统供应商沟通，以确保系统能够尽快恢复至正常状态。然而，通过使用本流程，基础设施团队应能够完成调试过程的初步步骤，尽可能多地收集数据，甚至完全避免提交技术支持请求。

\chapter{初始事件报告}

在管理 GPU 系统事件时，与所有系统事件一样，拥有一套设计良好的流程有助于更快速地诊断事件，使系统更快恢复生产状态。在任何事件发生之初，无论是系统自动检测到的还是用户报告的，都应尝试记录以下问题：

\begin{itemize}
  \item 观察到了什么异常现象？
  \item 事件是何时被发现的？
  \item 事件是如何被发现的？
  \item 该行为是否在多个系统或组件上均被观察到？
  \item 如果是，涉及多少系统？发生频率如何？
  \item 系统、驱动程序或应用程序的行为最近是否有任何变更？
\end{itemize}

收集这些问题的相关信息以启动调试流程非常重要，因为这能提供对问题的最佳理解。通过记录这些信息，可以将其与其他事件进行关联分析，从而更好地了解整体系统行为与健康状况。

\chapter{GPU 节点故障排查}

系统支持团队有多种途径可以获知基于 GPU 的系统上的潜在问题通知。这些通知可能来自错误报告、系统事件监控和诊断指标。虽然这些事件可能影响 GPU 系统的运行，但并非所有事件都需要系统供应商介入处理。此外，还有一些通用工具可用于在系统出现问题后收集数据，这些数据对本地系统支持团队和供应商进行节点故障排查都非常有用。

图\,\ref{fig:triage-flow} 至图\,\ref{fig:debug-app} 是基于所提供的信息（表明可能存在节点问题）启动节点故障排查的流程图。

\section{报告 GPU 问题}

在为系统供应商收集数据时，应包含以下信息：

\begin{itemize}
  \item 基本系统配置，如操作系统和驱动程序信息。
  \item 问题的清晰描述，包括描述问题的关键日志消息。
  \item 已执行的调试步骤列表。
  \item 上述关键消息所在日志的完整内容。
  \item \texttt{nvidia-bug-report.sh} 的输出。
  \item HGX 系统的 Fabric Manager 日志文件。
  \item DCGM 诊断日志。
  \item 如果与用户应用程序相关，应提供尽可能多的应用程序详细信息（ISV 代码、框架、版本号等）以及源代码链接（如有可能）。
  \item 向系统供应商提交工单。
\end{itemize}

\section{运行 DCGM 诊断}

\begin{quote}
\textbf{DCGM 是一个系统级工具，提供面向生产环境的诊断测试，用于评估节点健康状况和集群就绪性。} 详情请参阅 \url{https://docs.nvidia.com/datacenter/dcgm/latest/dcgm-user-guide/dcgm-diagnostics.html}。
\end{quote}

运行测试有许多可用选项，也有一些默认配置适用于大多数系统，通过 \texttt{-{}-run}（\texttt{-r}）选项指定。可用的测试套件包括 1（短）、2（中）和 3（长）。要对系统进行完整压力测试，请使用长测试（3）选项。

\begin{lstlisting}
# dcgmi diag -r 3
\end{lstlisting}

长测试大约需要 30 分钟。其他选项可用作预检或在系统 prolog 中验证节点，以确保其在开始作业之前处于正常状态。

当 DCGM 诊断发现问题时，应尝试解决该问题。配置相关的问题可通过 IT 命令处理，DCGM 可能会提供从何处开始的建议。如果诊断测试发现 GPU 或 NVSwitch Fabric（如存在）的运行存在问题，请检查节点及其配置是否有异常。

\section{运行现场诊断}

现场诊断（Field Diagnostic）是 NVIDIA 用于确定 GPU 健康状况的权威且全面的工具。通常在启动 RMA（返修授权）流程之前必须运行此工具。请联系您的系统供应商，了解何时、是否以及如何运行此工具。

\section{网络测试}

可以使用 NCCL 性能测试（\url{https://github.com/NVIDIA/nccl-tests}）来测试网络性能和延迟。特别是，\texttt{all\_reduce\_perf} 测试非常适合用于评估节点组之间的网络性能。节点间的成对测试应产生相同的性能结果。当识别出性能较慢的节点对时，应使用这些节点与其他不同节点重新测试以隔离问题。在多轨网络拓扑中，测试还可以隔离到特定轨道以确定哪个网络接口存在问题。更多信息请参阅 \url{https://developer.nvidia.com/nccl}。

建议在系统投入生产之前运行网络测试并记录所达到的性能。这将帮助您了解网络性能并为未来的比较提供基线。节点组的性能可能因系统网络拓扑而异。请与您的网络供应商/架构师协调，以了解应可达到的性能水平。

\section{调试应用程序}

调试应用程序通常需要将调试器附加到用户应用程序，并在应用程序崩溃前尽可能多地收集数据。导致应用程序崩溃的问题可能与系统软件有关，但调试过程是了解崩溃位置以及能否从应用程序端解决问题的最快方式。从调试器收集的数据可供 NVIDIA 用于开始调试系统软件问题。

CUDA-GDB 是 NVIDIA 用于调试在 Linux 和 QNX 上运行的 CUDA 应用程序的工具。它使开发人员能够调试应用程序，而无需受到模拟和仿真环境引入的潜在差异的影响。详情请参阅 \url{https://developer.nvidia.com/cuda-gdb} 和 \url{https://docs.nvidia.com/cuda/cuda-gdb/index.html}。

如果问题与性能相关，可以使用性能分析器（profiler），如 Nsight Systems 或 Nsight Compute，来了解应用程序的性能和瓶颈。系统软件的变更可能导致应用程序性能的变化，因此即使应用程序本身没有变更，对应用程序进行性能分析也是一个重要步骤。分析器数据可以突出显示 CPU、存储或 GPU 中的性能问题，帮助了解性能问题可能源自何处。

\section{收集节点指标}

节点性能和运行状态可以通过系统指标来推断。收集系统指标的时间序列数据可以帮助管理员检测节点何时开始出现异常运行，或用于主动式系统维护。

可以使用带内工具（如 DCGM Prometheus 插件）和带外工具（如 IPMI）来收集指标。有关 DCGM Prometheus 插件的详细信息，请参阅 \url{https://docs.nvidia.com/datacenter/cloud-native/gpu-telemetry/latest/kube-prometheus.html}。

DCGM 提供了面向 Kubernetes 部署的容器化版本。详情请参阅"在 Kubernetes 中使用 DCGM 监控 GPU"。

在基于 GPU 的系统上应收集的关键指标包括：

\begin{itemize}
  \item 各电源供应器的功耗
  \item PDU（配电单元）处的功耗
  \item PDU 功率因数
  \item CPU 功耗和 CPU 温度
  \item GPU 功耗、GPU 温度、GPU 显存温度和 GPU 时钟频率
  \item NVLink 指标（仅 A100）
  \item 节点风扇转速
  \item 进风口温度
\end{itemize}

根据数据中心的具体情况，其他指标也可能有用。

数据可以存储在时间序列数据库或其他 NoSQL 数据库中，并根据需要进行查询。此外，可以创建告警规则来检测可能对系统和应用程序性能产生负面影响的条件。这些告警包括：

\begin{itemize}
  \item 进风口温度上升可能表明数据中心冷却系统存在问题，或有杂物阻塞了气流。
  \item GPU 和 GPU 显存温度过高将导致 GPU 降频（throttling）和性能下降。
\end{itemize}

\section{在错误发生前捕获错误}

本文档的目标是帮助管理员发现最常见的系统问题，并提供解决方案或快速解决路径。然而，最好在系统问题影响用户作业之前就捕获它们。最佳时机是在作业开始时通过初始化容器（init-container）或 prolog 脚本执行检查，或在作业结束时通过 epilog 脚本执行检查。以下是一些建议的测试项目，有助于在系统错误影响用户作业之前捕获它们。

\vspace{0.5em}
\noindent\textbf{GPU 专项检查：}

\begin{itemize}
  \item 确认关键软件组件已加载
  \begin{itemize}
    \item GPU 驱动程序
    \item \texttt{nv-peer-mem}（用于 GPUDirect RDMA，\url{https://developer.nvidia.com/gpudirect}）
    \item Gdrdrv（用于 GDRCopy，\url{https://github.com/NVIDIA/gdrcopy}）
    \item HGX 系统的 Fabric Manager
  \end{itemize}
  \item 确认节点的特定 GPU 功能配置正确（GPU 数量、PCIe 链路速度、VBIOS 等）。
  \item 确认驱动程序持久化设置正确。驱动程序持久化应通过 Persistence Daemon 控制。（\url{https://docs.nvidia.com/deploy/driver-persistence/index.html}）
  \item 对于基于 NVSwitch 的系统：
  \begin{itemize}
    \item 确认 Fabric Manager 正在运行
    \item 确认 NVLink Fabric 拓扑正确
  \end{itemize}
  \item 检查是否有最近的 XID 错误
  \item 运行 \texttt{dcgmi diag -r 2} 以快速检查系统软件和 GPU 功能
\end{itemize}

\noindent\textbf{其他非 GPU 检查项目：}

\begin{itemize}
  \item 检查所有网络链路是否处于活动状态
  \begin{itemize}
    \item 这可以是简单的活动状态检查，也可以是更全面的检查，验证预期链路速度、固件版本、PCIe 链路宽度等。
  \end{itemize}
  \item 检查所有预期的文件系统挂载点是否存在
  \item 通过运行一些 CUDA 工作负载来检查节点设置是否正确。详情请参阅 \url{https://docs.nvidia.com/datacenter/tesla/hgx-software-guide/index.html} 和 \url{https://docs.nvidia.com/datacenter/tesla/driver-installation-guide/index.html}
\end{itemize}
