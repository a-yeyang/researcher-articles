\begin{abstract}
要实现“用 AI 智能体（agent）自动完成关键 IT 任务”这一愿景，离不开对所提出方案有效性的度量与理解能力。我们提出 ITBench，一个用于对 AI 智能体进行评测（benchmark）、使其能够应对真实世界 IT 自动化任务的框架，并为此提供一套系统化的方法论。我们的首个发布版本聚焦于三个关键领域：站点可靠性工程（SRE，Site Reliability Engineering）、合规与安全运营（CISO，Compliance and Security Operations）以及财务运营（FinOps，Financial Operations）。该框架的设计使 AI 研究者能够借助一键式（push-button）工作流和可解释的指标，理解 AI 智能体在 IT 自动化中所面临的挑战与机遇。ITBench 包含一组初始的 94 个真实世界场景（scenario），并可通过社区贡献轻松扩展。我们的实验结果表明：由当前最先进（state-of-the-art）模型驱动的智能体仅能解决 13.8\% 的 SRE 场景、25.2\% 的 CISO 场景以及 0\% 的 FinOps 场景。我们期望 ITBench 能成为推动 AI 驱动的 IT 自动化迈向正确、安全、快速的关键使能力量。
\end{abstract}

\section{引言}

现代 IT 系统正在驱动我们经济的方方面面。随着云计算和敏捷开发实践（agile development practices）的普及，这些系统的复杂度已显著上升（Harvard Business Review Research Report, 2022; Fuller, 2024）。对这些系统进行有效管理正变得极具挑战性：企业在努力跟上这种日益增长的复杂度。从首席信息官（CIO）到站点可靠性工程师，再到安全与合规人员以及一般 IT 工程师，各类 IT 角色（persona）都在为确保 IT 系统的韧性、可靠性、安全性以及成本高效运营而疲于应对。

近期的 CrowdStrike 大规模宕机事件凸显了这些挑战：它令我们社会中最关键的系统纷纷瘫痪，从医院服务到航空旅行无一幸免，估计给 500 家美国财富（Fortune 500）公司造成了高达 54 亿美元的损失（Kerner, 2024）。该事件再次说明，智能化的 IT 事件处置、合规与风险管理能力至关重要——这一议题在欧洲的《数字运营韧性法案》（DORA，Digital Operational Resiliency Act）中亦有论及（Parliament and the Council of the European Union, 2024）。

AI 智能体的日益流行，以及人们对其处理复杂任务能力的预期，已经推高了对“用 AI 智能体管理 IT 系统”的需求（John, 2024; Miguel Carreon, 2024; Pujar et al., 2023）。鉴于 IT 任务的复杂性，这一研究方向面临的一大障碍在于：如何建立系统化的方法，以在 AI 智能体投入生产部署之前评估其有效性（Bogin et al., 2024; Kapoor et al., 2024）。因此，亟需开发一种基于真实 IT 任务及其对应环境来评测 AI 智能体的方法。

本文正是针对这一关键需求，提出了 ITBench——首个兼具全面性与前瞻性、用于对真实生活中 IT 自动化任务进行评测的框架。ITBench 的目标是度量 AI 智能体在各类复杂且真实的 IT 任务上的表现，覆盖多种角色，包括：聚焦可用性与韧性的\emph{站点可靠性工程（SRE）}、确保 IT 实现合规与安全的\emph{合规与安全运营（CISO）}、以及强制实现成本效率并优化投资回报的\emph{财务运营（FinOps）}等（如图~\ref{fig:itbench-personas} 所示）。

ITBench 旨在推动该领域的创新并确立新的标准。我们的贡献可沿以下三条主线归纳：

\begin{itemize}
  \item \textbf{反映真实世界：} ITBench 针对的是在生产环境中具有现实意义且普遍存在的 IT 自动化需求。其中 SRE 场景基于我们自有 SaaS 产品中观察到的真实事件；CISO 场景基于 CIS 基准（面向互联网安全，CIS）；FinOps 场景则由 FinOps 基金会（FinOps Foundation）通过关键业务成果加以界定（Foundation, 2025a）。
  \item \textbf{开放且可扩展，覆盖全面的 IT 任务：} 我们将 ITBench 视为一个中枢平台，用于在多样化的 IT 自动化用例中评测 AI 驱动的解决方案。为此，我们提供 IT 评测套件以及支持纵向扩展（即增加更多场景）和横向扩展（即增加更多角色）的框架，从而实现对 IT 任务的广泛覆盖。ITBench 是一个基于开源技术构建的开源框架，同时也允许各组织使用专有技术，以开发和评测其自有解决方案。
  \item \textbf{支持带部分评分的自动化评估：} ITBench 旨在提供建设性反馈，以推动针对 IT 问题的智能体（agentic）解决方案在设计上的改进。它包含一套全面的评估框架与排行榜（leaderboard），可在智能体推理过程的各个阶段向用户提供有效反馈。
\end{itemize}

ITBench 提供一键式部署以及用于搭建环境、运行时智能体、护栏引擎（guardrail engine）、授权与认证的工具。它使开发者和研究者能够构建用于管理复杂 IT 系统的新型解决方案。目前，ITBench 处理的是反应式（reactive）问题，包括事件的诊断与处置、受监管环境中针对新管控规则的合规评估、以及成本管理事件。未来，我们计划扩展评测基准的评估能力，并为 IT 流程的更多方面纳入新的评测基准。当前，ITBench 由一组初始的 94 个场景构成，覆盖 SRE（42 个）、CISO（50 个）和 FinOps（2 个），其对应的成功场景处理率分别为 13.8\%、25.2\% 和 0\%（详见第~\ref{sec:related-work} 节）。

我们相信，正如极具影响力的 SWEBench（Jimenez et al., 2024）一样，我们这一全新的 ITBench 框架——它封装并度量了 AI 智能体自动化处理复杂、真实 IT 任务的能力——将同样推动真实世界 IT AI 智能体性能的可观加速进步。

\section{相关工作}
\label{sec:related-work}

ITBench 面向 IT 自动化中广泛角色的一整套全面任务。ITBench 的首个发布版本聚焦于评估 IT 运维（ITOps）范畴内的场景。图~\ref{fig:itbench-personas} 展示了当前所针对的角色及其例行面对的示例任务。当前，针对 ITOps 中 SRE、CISO 和 FinOps 的评测基准开发呈现出明显的上升趋势与关注热度。

TrainTicket（Zhou et al., 2018）通过对真实世界事件的工业调研收集了 22 个场景，并在 TrainTicket 应用中使用硬编码故障以聚焦于故障定位。AIOpsLab（Chen et al., 2024a）提供了 10 个聚焦 SRE 的场景（在该工作中称为 problems），借助真实环境（系统）集成，允许对任务、时间序列和表格数据进行交互式访问。InsightBench（Sahu et al., 2024）提供了 100 个场景，利用静态表格数据和合成场景对工单（ticket）数据进行分析。TSB-AD（Liu and Paparrizos, 2024a）聚焦于异常检测，包含 40 个合成场景。

CIS-Benchmark（CIS, 2024）提供了用于保障 IT 基础设施安全的最佳实践。尽管名称中带有“benchmark”一词，但它仅提供推荐策略，并不提供实验平台。近期，云原生计算基金会（CNCF）的沙箱项目 OSCAL-compass（OSCAL-compass, 2024）发布了一套 SDK，用于支持将 CIS 人类可读格式转换为美国国家标准与技术研究院（NIST）的 OSCAL（OSCAL, 2024）合规标准，以便在合规自动化中进行编程化使用。ITBench 的 CISO 自动化正是利用了这一技术来评估策略要求。

FinOps 基金会（Foundation, 2025a）提供了评测基准，用于跨组织和部门比较云财务表现，聚焦于资源利用效率、合同覆盖率、成本分摊等关键绩效指标（KPI）。这些评测基准通过评估内部与外部指标，帮助评估云使用效率，从而促进结构化、协作式的云优化方法。

尽管现有评测基准对于特定任务和用例而言是宝贵的资源，并凸显了系统化评测的迫切需求，但它们在反映真实世界 IT 问题、覆盖广泛的 IT 全景以及自动化评估方面仍有局限。这些局限在 ITBench 中得到了解决，如表~\ref{tab:benchmark-comparison} 所示。

\section{ITBench}

ITBench 是一套系统化的评测框架与运行时环境，旨在评估被赋予 IT 运维自动化任务的智能体（agent）。它包含一套健壮的架构（见图~\ref{fig:itbench-framework}），由 AI 智能体（AI Agent）、场景规约与环境（Scenario Specification and Environment）、评估器（Evaluator）和排行榜（Leaderboard，leaderboard）组成，从而支持全面的性能评估。

在此，我们对几个关键组件做简要概述：1）场景规约与环境，2）AI 智能体，3）排行榜。更多细节见附录 B。

\subsection{场景规约与环境}

该评测基准（benchmark）纳入了一组我们称之为\emph{场景}（scenario）的问题集合。例如，ITBench 中的一个问题是要在 Kubernetes 环境中解决“服务订单管理（service order-management）的高错误率”。另一个与 CISO（合规与安全运营，Compliance and Security Operations）角色相关的例子是评估“针对 RHEL 9 检测到的新控制规则”的合规态势。一个根本性的挑战在于，如何在一个可管理的测试床（testbed）环境中复现这类问题。一个场景就是一个具体问题（或多个问题）发生于其中的可运行测试床。

一个场景 $p$ 通常对应于 ITBench 中待解决的一个问题。我们将 $p$ 形式化为一个四元组 $<M, E, T, D>$，其中各变量含义如下：

\paragraph{场景规约（Scenario Specification）。} $M$ 表示每个场景的元数据（metadata）与部署描述符，存储于 ITBench 的场景规约数据库（Scenario Specs database）中（见图~\ref{fig:itbench-framework}）。每个场景的示例元数据元素包括 \emph{scenario\_name}、\emph{scenario\_description}、\emph{scenario\_domain}、\emph{scenario\_class}、\emph{scenario\_complexity} 和 \emph{scenario\_groundtruth}（见表~\ref{tab:exemplar-scenarios}），其定义如下：

\begin{itemize}
\item \emph{scenario\_name} 是赋予某个场景的名称。例如，ITBench 中的一个场景名为“Recommendation Service Cache”。
\item \emph{scenario\_description} 描述该场景。例如，某场景的描述为“Recommendation Service in Astronomy Shop has a cache failure（天文商店中的推荐服务发生缓存故障）”。
\item \emph{scenario\_domain} 表示不同的角色（persona），即 IT 自动化范畴内的 “SRE”“CISO”“FinOps”。
\item \emph{scenario\_class} 用于对相似场景进行分组，例如 “Kyverno-opa”“Kyverno-update”“CacheFailure”“HighCPU” 与 “CorruptImage”。
\item \emph{scenario\_complexity} 刻画一个问题的难度，并依据领域知识来定义。图~\ref{fig:itbench-scenarios}a 给出了基准中 SRE、CISO 与 FinOps 场景的占比分布。图~\ref{fig:itbench-scenarios}b、\ref{fig:itbench-scenarios}c 和 \ref{fig:itbench-scenarios}d 分别展示了 SRE、CISO 与 FinOps 的 \emph{scenario\_complexity} 分布。SRE 场景基于我们自有 SaaS 产品中观测到的真实事件开发而成。CISO 场景基于 CIS 评测基准构建（for Internet Security, CIS）。由于缺乏标准化基准，FinOps 场景在 ITBench 中占比较少。我们的 FinOps 场景基于 FinOps Foundation（Foundation, 2025a）所确定的“领域（Domains）”与“能力（Capabilities）”来描述关键业务成果。
\item \emph{scenario\_groundtruth} 记录与具体任务相关的结果，评估器（Evaluator）用它来与智能体的预期输出进行比对。例如，对于 SRE 的事件解决而言，诊断（diagnosis）任务的真值（ground truth）包含故障传播链（fault propagation chain）所涉及的实体列表、实际的故障传播链以及故障条件；而对于缓解/修复（mitigation）任务，则记录可能的缓解动作。
\end{itemize}

\paragraph{环境（Environment）。} $E$ 表示问题发生于其中的可运行测试床。环境内的组件通过暴露 API 来观测并控制该环境。当智能体构建者（Agent Builder）将智能体注册以参与评测时，评测基准运行器（Benchmark Runner，见图~\ref{fig:itbench-framework}）会随机选取一组场景，并可依据 \emph{agent\_type} 与 \emph{agent\_level} 对其进行可选的筛选。随后，评测基准运行器遍历该场景集合，并为每个场景实例化一个测试床。环境的一个例子是：安装了 OpenTelemetry Astronomy Shop Demo 应用（Community, 2024）的 Kubernetes 集群，其可观测性（observability）栈包括 Grafana（gra）、Loki（lok）、Jaeger（jae）和 Prometheus（pro），并配有在环境中诱发问题的机制。

\paragraph{触发事件（Triggering Events）。} $T$ 是一组因环境中某个特定问题的具体表现而发生的触发事件。各类工具被配置用于观测环境，并在出现异常状况时引发触发事件。一个触发事件的例子是“High Error Rate on adservice（adservice 上的高错误率）”，它可能因缓存故障问题而在环境中被触发。

\paragraph{期望结果（Desired Outcome）。} $D$ 定义了自动化目标，代表最终目的。例如，在 SRE 事件解决的情形下，最终目标是在 $E$ 中清除 $T$。

\subsection{AI 智能体}

\begin{figure}[H]
\centering
\includegraphics[width=0.9\linewidth]{fig03.pdf}
\caption{图 3：作为 POMDP 的智能体与环境。智能体通过 ITBench 工具箱（toolbox）所暴露的 API 与环境交互。}
\label{fig:agent-pomdp}
\end{figure}

在 IT 自动化中，不同角色聚焦于各自特定的期望结果，由此定义其自动化目标。对于 SRE 而言，事件解决（incident resolution）是首要目标。实现这一目标可能涉及多个步骤，例如诊断一起事件，也可能是单一步骤，例如生成一份诊断报告。CISO 角色聚焦于法规控制态势评估过程，包括 \emph{Collect evidence（收集证据）} 与 \emph{Scan assessment posture（扫描评估态势）} 任务。FinOps 角色聚焦于成本管理，其示例任务包括 \emph{Identify inefficiency（识别低效）} 与 \emph{Mitigate inefficiency（缓解低效）}。在评估过程中，每一步（任务）都被独立评估，并使用明确定义的指标进行度量，见表~\ref{tab:personas-tasks}。

ITBench 的目标是在广泛的真实 IT 自动化任务上评估智能体，这些任务原本由 SRE、FinOps、CISO 等角色来执行。

在本文中，一个 AI \emph{智能体}（agent）被定义为一个自主或半自主的软件程序，它使用大语言模型（LLM）进行规划、决策、与目标环境交互，并执行动作以达成目标。一个 AI 智能体应当能够通过与环境交互，成功处理 ITBench 中的任意场景。

如图~\ref{fig:agent-pomdp} 所示，智能体与环境构成一个部分可观测马尔可夫决策过程（POMDP），其中状态是环境的快照。状态转移由环境决定，随后被智能体（部分地）观测到。

给定一个在环境 $E$ 中实例化的场景 $p$，智能体通过工具之一探测环境，并接收到一个观测 $o_t \in \mathcal{O}$，据此决定下一个动作：

\begin{equation}
a_t = f(o_t \mid \bar o_{t-1}; \bar a_{t-1}) \tag{1}
\end{equation}

此处 $f$ 是智能体的决策函数。$\bar o_{t-1}$ 是直到时刻 $t-1$ 为止的观测序列，$\bar a_{t-1}$ 是直到 $t-1$ 为止所采取的动作序列。

在初始时刻，$o_0$ 可以是一个展示环境问题状态的触发事件。给定状态 $s_{t-1}$ 与动作 $a_{t-1}$，环境转移到下一个状态：

\begin{equation}
s_t = g(s_{t-1}, a_{t-1}) \tag{2}
\end{equation}

观测 $o_t$ 被确定为状态的函数，且一般而言是环境状态 $s_t$ 的代理（proxy），因此

\begin{equation}
o_t = h(s_t) \tag{3}
\end{equation}

由此，整个表述可被视为一个 POMDP。动作集合 $\mathcal{A}$ 定义为 $\mathcal{Q} \bigcup \{\perp\}$，其中 $\mathcal{Q}$ 是工具集合，$\perp$ 表示智能体的“停止动作（stop action）”。我们将 $t^*$ 定义为智能体停止的时刻：

\begin{equation}
t^* = \min\{t \mid a_t = \perp\} \tag{4}
\end{equation}

智能体对结果进行反思（reflect），以指导其下一个动作，并持续进行直到最终目标达成。给定一组智能体所处理的场景，其目标是最大化按如下方式定义的成功率：

\begin{equation}
\mathbb{E}_{p\sim\pi_p}\big( \mathbb{I}( g(s_{p,t^*}, f(o_{t^*}\mid \bar o_{t^*-1}, \bar a_{t^*-1})) = s_{p,G}) \big) \tag{5}
\end{equation}

其中 $\mathbb{I}$ 是将终止状态与目标状态进行比较的指示函数（indicator function），$\pi$ 是场景的分布。

\subsection{基线 AI 智能体}

我们开发了多个基线智能体（baseline agent）：面向 SRE 的 SRE-Agent、面向 CISO 的 Compliance Assessment Agent，以及面向 FinOps 的 FinOps-agent。这些智能体均采用了最先进的智能体技术（agentic techniques），例如基于 ReAct 的规划（Yao et al., 2023）、反思（reflection）（Shinn et al., 2023）以及任务分解（disaggregation）（Xu et al., 2023）。反思技术则各有不同，涵盖语法检查/代码风格检查（Shinn et al., 2023）、语义校验（Xie et al., 2024a）以及 llm-as-a-judge（大模型充当评判者）（Zheng et al., 2023）。

我们随 ITBench 开源了两个基线智能体（SRE\footnote{\url{https://github.com/IBM/itbench-sre-agent}} 与 CISO\footnote{\url{https://github.com/IBM/itbench-ciso-caa-agent}}）。我们使用开源的 CrewAI（cre）框架来创建和管理智能体。这些智能体可被配置为通过 watsonx、Azure 或 vLLM 使用各种 LLM。每个智能体在初始化时都会获得一个提示词（prompt），该提示词描述其目标、上下文、任务以及预期输出格式。其中包含上下文学习示例（in-context learning examples），用以引导智能体并演示工具用法。智能体使用自然语言来访问工具，从而与环境交互以收集信息。

诊断过程中收集到的日志（logs）、追踪（traces）与指标（metrics）数据量庞大，会淹没当前任何可用 LLM 的上下文窗口。因此，面向 SRE 或 FinOps 角色的智能体配备了专门的工具来与环境交互（参见图~\ref{fig:agent-pomdp}）：1）NL2Traces，以结构化格式提取追踪数据；2）NL2Metrics，分析关键系统指标；3）NL2Logs，高效解析日志数据；4）NL2Kubectl，执行 Kubernetes 特定的操作；此外还有一个摘要工具，用于将海量数据浓缩为可操作的洞见。例如，智能体可使用 NL2Kubectl 工具来“列出 default 命名空间中的所有 Pod”。NL2Kubectl 工具继而使用一个 LLM 将该自然语言表述转换为可执行命令，即 “kubectl get pods -n default”。

类似地，新法规与技术所需的合规评估若直接交给 LLM，其证据与多样的策略语言（policy language）也会令其不堪重负。为 CISO 合规评估自动化所设计的合规智能体（compliance agent）配备了专门的工具。这些工具具备以下能力：1）从自然语言规约出发生成策略，例如 Kyverno 或 OPA Rego 的策略即代码（Policy as Code）；2）生成用于收集证据的脚本；3）访问代码仓库（例如 git）以支持面向代码管理的 GitOps 工作流；4）部署并执行所生成的策略，以完成评估任务。

\subsection{排行榜}

ITBench 包含一个排行榜（leaderboard），以促进可复现性与对比分析，其遵循 AI 通用任务框架（AI common task framework）（Donoho, 2019; Varshney et al., 2019）。该排行榜提供了一套预定义、可扩展的性能指标，旨在就智能体相对于评估标准的表现给出清晰洞见。

ITBench 设计了\emph{针对部分正确解的评分方法（scoring methods for partially correct solutions）}，以便为总结性评估（summative assessment）提供有意义的反馈。这一全面的方法为评估和推进 IT 自动化中的 AI 驱动解决方案确立了新标准。对于智能体所处理的每个场景，在任务完成时，ITBench 会记录最终的系统状态，随后在所有场景运行结束时，将其与预定义的真值数据一同用于验证智能体在所有场景上的表现优劣。对于智能体所处理的每个场景，在任务完成时，ITBench 都会记录最终的系统状态，随后在所有场景运行结束时，将其与预定义的真值数据一同用于验证智能体在所有场景上的表现优劣。

我们正开源 ITBench\footnote{\url{https://github.com/IBM/itbench-sample-scenarios}} 场景中的一个小子集（94 个中的 11 个）以及基线智能体，以帮助社区通过实际示例熟悉 ITBench。我们保留 ITBench 中其余场景，用于对所提交的智能体解决方案进行评测与评估。
\section{实验结果}

\subsection{评测设置}

为了理解大语言模型（LLM）的推理与规划能力对 ITBench 各场景的影响，我们用不同的 LLM 实例化我们的智能体（agent），同时覆盖自然语言推理与代码生成两类任务。具体而言，对于依赖自然语言理解与推理的任务，我们采用 GPT-4o\footnote{检查点版本 2024-11-20}、Llama-3.3-70B-instruct、Llama-3.1-8B-instruct 和 Granite-3.1-8B-instruct；对于以代码为核心的用例，我们采用 GPT-4o-mini、Llama-3.1-405b-instruct 和 Mixtral-8x7b-instruct。所有模型均使用 128K tokens 的上下文窗口，使其能够处理较长的输入序列。

我们的实验主要在 AWS EC2 实例（m4.xlarge）上进行，不过 ITBench 也可以借助伪集群（pseudo-cluster）方便地部署在消费级笔记本电脑上，从而更便于开发 AI 智能体（附录 C.4.1）。

下面我们概要介绍基线智能体在 ITBench 的 SRE、CISO 与 FinOps 场景上的表现。我们的发现表明，无论是开源模型还是专有模型，在真实世界任务上往往都举步维艰，这凸显了构建能够推动基础模型推理与规划能力极限的评测基准（benchmark）的重要性。更全面的结果与逐场景的详细讨论，请分别参见附录 C（SRE）、附录 D（CISO）与附录 E（FinOps）。

\subsection{总体结果}

表~\ref{tab:itb-sre}、表~\ref{tab:itb-ciso} 与表~\ref{tab:itb-finops} 分别展示了 SRE-agent、CISO-agent 与 FinOps-agent 的表现。

\paragraph{SRE。} 我们以 SRE-Agent 诊断与缓解生产事故（例如“前端服务出现高错误率”）的能力来衡量其效率。

诊断效率用 pass@1 来衡量 (Chen et al., 2021)（即是否识别出标准答案中所述的根因），用 NTAM（归一化拓扑感知指标，Normalized Topology-Aware Metric）来衡量根因与故障传播链（fault propagation chain），并测量诊断耗时\footnote{NTAM 是一种归一化拓扑感知指标，借助系统与应用的拓扑结构来衡量所预测的根因与故障传播链的质量。详见附录 C.6.3。}。缓解效率则以 pass@1（即告警是否被清除）与平均修复时间来衡量。

如表~\ref{tab:itb-sre} 所示，在所有 SRE 场景中，GPT-4o 始终优于其他模型，在诊断（13.81\%）与缓解（11.43\%）上均取得最高的 pass@1 分数，并在 NTAM（FL 与 FPC）指标上取得最高分。Llama-3.3-70B 总体排名第二，在多数指标上仅次于 GPT-4o。8B 量级的模型缓解成功率较低。出乎意料的是，Granite-3.1-8B（未经任何专门微调）在诊断任务上取得了高于 Llama-3.1-70B 的准确率。

移除跟踪（trace）数据会大幅降低成功率（参见附录中的表 19 与表 20）。例如，GPT-4o 的诊断 pass@1 从有跟踪数据时的 13.81\% 跌至无跟踪数据时的 9.52\%，缓解成功率更骤降至 2.86\%。这凸显了系统可观测性（observability）在 SRE 中的关键作用，而 ITBench 能够在不同条件下对此加以评测。鉴于实践中并不存在完美的可观测性，如何引导 SRE-agent 去采集新的可观测性数据，以及如何帮助 SRE-agent 在不完全可观测的情况下对故障进行推理，是一个重要但尚待解决的问题。

\paragraph{CISO。} 我们衡量智能体在表~\ref{tab:itb-scenarios} 中引入的四个场景类（scenario class）上的效能。每个场景类（\textit{scenario\_class}）施加一组不同的 CIS-benchmarks 要求（例如“尽量减少允许共享主机网络命名空间的容器数量”），每一类具有特定的复杂度级别（如易、中、难），并生成场景特定的代码工件（code artifact）。

CISO-agent 的效能基于以下能力来衡量：在场景类的多样环境中检测工件配置错误（即非合规，如不存在共享命名空间的容器最小数量要求，或计数超过阈值），或在随机注入了配置错误的环境中确认正确配置（即合规）。值得注意的是，基于 GPT 的模型在 pass@1 与处理时间（Time to Process）两项指标上均占据主导地位。其 pass@1 几乎比次优模型高出近 2 倍（在 llama-3.1-405b-instruct 与 mistral-large-2 之间交替），而 TTP 表明 GPT-4o 能在我们各场景类中以最短时间处理这些场景。

\paragraph{FinOps。} 我们以 FinOps-agent 诊断与缓解成本告警来源（例如“成本上升 20\%”）的能力来衡量其表现。诊断有效性用 pass@1（即是否识别出原因）来衡量。缓解有效性则以与最优运行成本的比例接近度（proportional proximity），以及该工作负载可达到的效率来衡量。

GPT-4o 始终优于所有其他模型，在诊断成本上升告警的来源上取得 33\% 的 pass@1 通过率。在与成本及工作负载效率相关的其他指标上，各模型的表现大体相当，没有任何模型能够达到最优的 CPU 与内存成本或交付出高 CPU 效率。

\subsection{场景复杂度的影响}

\paragraph{SRE。} 我们依据故障传播链长度、解决步骤数量、所涉技术多样性等因素，将场景划分为易、中、难，如公式 (6) 所述。我们的结果表明，随着场景复杂度（\textit{scenario\_complexity}）的提升，成功率（pass@1）明显下降。例如，GPT-4o（表现最佳的模型）对易、中、难场景的诊断成功率分别仅为 36\%、7.73\% 与 5.0\%（参见表 17）。同样，GPT-4o（表现最佳的模型）对易、中、难场景的缓解成功率分别仅为 21\%、12.27\% 与 0.0\%（参见表 18）。

没有任何模型能够缓解任一次运行中的难场景，而超过半数的易场景缓解是成功的。值得注意的是，GPT-4o 是唯一能够成功诊断多个“难”场景的模型。

\paragraph{CISO。} CISO 场景的复杂度与场景类直接对应。例如，Kyverno 场景的场景复杂度（\textit{scenario\_complexity}）为易，k8s-opa 与 rhel-opa 场景为中，而 kyverno-update 场景为难。正如预期，随着场景难度从易的 \textit{kyverno} 类提升至难的 \textit{kyverno-update} 类，所有模型都表现吃力。

\paragraph{FinOps。} 目前 ITBench 仅有两个 FinOps 场景，其中一个的场景复杂度为易，另一个为难。除 GPT-4o 外没有任何模型能够诊断（更无法缓解）难场景。

ITBench 中这一复杂度谱系确保了评测既能覆盖直观简单的问题，也能覆盖跨各类角色（persona）的高度复杂问题。

\subsection{环境固有的非确定性}

GPT-4o 在所有受评角色（SRE、CISO 与 FinOps）中仍是表现最佳者，但它在场景结果上仍表现出显著的波动性。例如，尽管已通过超参数调整力求一致性，搭载 GPT-4o 的 SRE-agent 仍难以保持确定性行为。搭载 GPT-4o 的 SRE-agent 在场景 13 的 10 次运行中仅有 6 次成功诊断出问题，场景 8 为 10 次中 1 次，场景 21 为 10 次中 8 次（所有场景的细节见图 14）。同样，它在场景 16 的 10 次运行中缓解了 6 次，场景 8 为 10 次中 2 次，场景 21 为 10 次中 5 次（所有场景的细节见图 17）。在 FinOps 与 CISO 场景中也观测到了这种固有的非确定性。

这些波动源于实时遥测（telemetry）数据的微小变化，以及大语言模型逐 token 生成的特性。通过在多次运行中跟踪这种动态行为，ITBench 为评估每个智能体的鲁棒性与可靠性提供了关键洞见。

\section{讨论与结论}

我们提出了 ITBench——首个用于评测 IT 自动化任务中 AI 智能体的框架与实验平台。ITBench 力求刻画现代 IT 系统的复杂性与 IT 任务的多样性。ITBench 的可复现性确保了这一社区驱动的工作能够克服大规模 IT 系统固有的非确定性。

ITBench 的关键设计原则之一是确保其灵活性，以支持不同 IT 系统的多样领域，并具备向新场景扩展的能力。尽管 ITBench 当前的覆盖范围已较为全面且具代表性，我们仍计划通过加入对现代 IT 自动化至关重要的其他流程，进一步丰富评测套件。此外，我们计划将评测从事件触发（event-triggered）型场景进一步扩展。我们正积极扩大对所支持流程的场景覆盖，并通过开放社区贡献来促进发展。我们邀请社区借助 ITBench，在合成沙箱环境中复现其受真实世界启发的事故。我们期望每一位贡献者都能将各自的专长带入这一领域。

我们期望 ITBench 能够推动基于 AI 智能体的技术创新，并对当今 IT 基础设施的安全、效率与智能产生直接影响。借助 ITBench，我们开始探索诸多深刻而激动人心的开放问题：如何开发专精于特定类型 IT 任务的领域专用 AI 智能体？如何编排具备多种专长的多个智能体，使其协作完成更大的项目？如何确保 AI 驱动方案的安全性？在开发多样的自适应智能体时，如何有效利用人在回路（human-in-the-loop）？我们邀请所有人参与回答这些问题，共同实现用 AI 智能体自动化关键 IT 任务的愿景。

\section{声明}

\subsection{伦理与更广泛影响}

本研究提出了一个新颖的评测框架，用于衡量 AI 智能体在多种复杂的真实 IT 任务上的表现，这有望成为推动正确、安全、快速的 AI 驱动 IT 自动化的关键支撑。尽管主要目标在于推进机器学习领域的发展，但由于本工作是一个构建于开源技术之上的开源框架，它使得拥有专有技术的组织能够更有效地利用它来开发和评测自身的解决方案；它也促进了社区内的思想共享，并降低了 IT 领域创新的门槛。

与系统交互的智能体带来若干风险。我们识别出在构建和使用 ITBench 及相关智能体时可能出现的三大主要风险，随后讨论我们为缓解这些问题所采取的措施。

第一是在系统上执行 LM 生成的命令/指令所带来的安全风险。例如执行 \texttt{kubectl delete node} 与 \texttt{rm -rf asset/} 这类命令。为防范这一点，我们将智能体容器化，并提供一个自包含的 Kubernetes 环境来创建各类场景。

第二，如果更广泛的社区对 ITBench 及相关智能体产生兴趣并在其基础上构建工作，那么也可能出现非法的评测数据集或基础设施被用来向受测设备注入恶意代码、或注入生成恶意代码的指令的情况。例如，一个声称为 ITBench 及相关智能体托管推理/评测框架的非官方代码仓库，可能包含这样一个任务实例：其问题描述指示 LM 智能体构建密钥日志记录功能并将其存储在隐藏文件夹中。为消除混淆并降低此类事件发生的可能性，我们在 GitHub 仓库、数据存储与网站上提供了清晰的指引，标明我们积极维护的官方仓库与渠道。我们也鼓励第三方将任何改进纳入我们的代码库，并协助整合此类贡献。

最后是 ITBench 智能体在现实世界中被部署所带来的后果。先前工作已对可执行攻击性安全措施的系统进行了概念化并提出原型。同样不难设想，像 SRE-Agent 这样的系统可能被纳入流水线，从而在上述用例中生成恶意代码与库。智能体在 ITBench 上的强劲表现意味着，未来的 AI 系统在上述用例中很可能日益娴熟。将 ITBench 智能体作为开源智能体发布，能够支持有关“为软件工程智能体设计何种健全有效的约束”这一方向的研究。它也可以作为一个系统，供法律专家与政策制定主体进行试验，以共同塑造端到端 AI 驱动软件工程的未来形态。

\subsection{可复现性}

为帮助更广泛的社区复现本文呈现的结果并在其基础上构建工作，我们将本项目创建的所有资源开源。交互式流水线的源代码、上下文管理逻辑、命令实现、接口设计以及其他一切，均已完整发布于一个 GitHub 仓库中。我们提供了大量文字与视频文档，说明如何运行和修改代码库的各个部分。从业者应能通过简单的脚本运行智能体，从而轻松复现我们的发现。本文正文与补充材料中呈现的结果，均可遵循仓库中的说明完整复现。最后，我们还维护着一个活跃的在线帮助论坛，以协助解决任何复现问题或关于如何在 ITBench 上构建的疑问。

\section{附录 A：相关工作（扩展）}

\paragraph{用于解决 IT 自动化任务的语言模型与智能体。}
人工智能/机器学习（AI/ML）正越来越多地用于处理 IT 自动化任务。下面按各类角色分别介绍相关工作。

\subsection{A.1 站点可靠性工程（Site Reliability Engineering）}

IT 场景\footnote{我们以宽泛的含义使用“场景（scenario）”一词，用以指代故障、性能问题、合规问题以及成本异常。}的解决涵盖了检测（例如，识别异常或服务中断）(Guo et al., 2015; Leners et al., 2011; Sigelman et al., 2010; Fonseca et al., 2007)、诊断（例如，通过指标与日志定位根因）(Tan et al., 2019; Jha et al., 2020; Ma et al., 2014; Salesforce, 2023) 以及缓解/修复（例如，运维层面的修复或代码变更）等任务。这些工作往往依赖于若干支撑性任务，例如工单分析与路由 (Gao et al., 2020; Liu et al., 2023b; Arzani et al., 2016)、异常检测 (Ashok et al., 2024; Chakraborty et al., 2023; Pham et al., 2024; Yao et al., 2024)、拓扑抽取 (Liu and Paparrizos, 2024a)、因果分析 (Budhathoki et al., 2022; Microsoft and contributors, 2023; Ikram et al., 2022; Chakrabarti et al., 2023) 以及干预性分析 (Wang et al., 2023; Bagehorn et al., 2022)。显然，这一领域已有大量研究，但由于真实世界系统的复杂性、故障的多变性，以及将上下文知识融入 AI 系统的挑战 (Jha et al., 2020)，全自动地解决故障或向人类提供可执行的洞见仍然难以实现。

大语言模型（LLM）的近期进展推动了其被用于工单数据分析与诊断任务 (Roy et al., 2024; Ahmed et al., 2023a; Xie et al., 2024b; Chen et al., 2023; Zhang et al., 2024)。其中最具代表性的例子包括用 LLM 进行因果图构建的 Cloud Atlas (Xie et al., 2024b)，以及用于工单分析、旨在诊断与缓解事件的 RCACopilot (Chen et al., 2023)。然而，与其他技术相比，它们的表现欠佳。例如，(Roy et al., 2024) 表明思维链（chain-of-thought）方法的准确率仅为 35\%。更近期的工作中，LLM 被用于智能体式（agentic）工作流，借助一系列真实或虚拟环境中的工具来完成诸如网页导航 (Drouin et al., 2024; Boisvert et al., 2024; Koh et al., 2024)、系统控制 (Sahu et al., 2024; Zhang et al., 2024; Chen et al., 2024a) 以及代码生成 (Yang et al., 2024a) 等任务。然而，这些工作的初步结果在成功率上表现出很高的方差——从 InsightBench (Boisvert et al., 2024) 中的 35\%，到 Flash (Chen et al., 2024a) 中用于事件解决的基于 ReAct 的智能体（用于工单数据分析 (Roy et al., 2024)）所达到的 100\%，尽管后者本应是一项远比在表格与工单数据中发现既有洞见更困难的任务。我们自己的研究结果同样表明，LLM 与智能体在持续稳定地完成事件解决任务方面存在困难。\textit{我们断言，成功率上的方差之所以存在，是因为这些数据集在现实性方面存在差异。}这凸显出业界迫切需要标准化、开源的评测基准（benchmark），以有效评估并改进 AI 方法在事件解决任务上的效能。

\paragraph{面向 SRE 的评测基准。}
IT 运维（ITOps）任务的评测基准生态仍处于早期阶段，仅有少数现有工作针对该领域的特定方面展开。AIOpsLab (Chen et al., 2024a) 聚焦于解决在真实环境中创建的、涵盖九个不同问题的 IT 事件。但它并未遵循 SRE 在系统与应用可观测性（observability）方面的最佳实践，例如它使用了一套告警管理系统，缺乏全面的覆盖度分析，也没有用于系统化自动评测的排行榜（leaderboard）。

InsightBench (Sahu et al., 2024) 针对 ServiceNow 工单数据的分析——这是事件路由与查找相关历史事件的一项关键支撑任务——但其对合成数据的依赖以及缺乏真实环境，限制了它在智能体式工作流中的适用性。类似地，TSB-AD (Liu and Paparrizos, 2024b) 是为单变量与多变量异常检测（事件检测的一项核心任务）而设计的。然而，它仅限于合成数据，且只聚焦于异常检测。

\subsection{A.2 合规（Compliance）}

合规自动化软件正在兴起，旨在帮助企业简化并自动化合规流程，减少人工监控与法规追踪的需求。这确保了对法律法规的持续遵守。其中，合规即代码（compliance as code）是 IT 行业中一项非常新近的进展，其推动力来自企业以及审计机构正从年度审计转向期望对受监管环境的合规态势与遭受网络攻击的风险进行持续、自动化的度量与管控。

近期工作 (Papanikolaou et al., 2011; Tupsamudre et al., 2022) 已应用 AI/ML 技术来加速这些任务，重点是将监管要求映射到诸如 NIST 800-53 (NIST 800-53) 之类的标准控制框架上。我们在当前 ITBench 方案中的智能体式自动化，开创性地借助 AI/ML 来撰写合规制品，通过将合规即代码桥接为策略即代码（policy as code）来实现这一目标。相较于合规即代码，策略引擎在 IT 行业中已有较长的应用历史；然而，诸如 (Int, d) 这样新兴的通用型策略引擎试图解决业界对一个统一的持续合规框架的需求。据我们所知，目前尚无任何工作——尽管这是关键且亟需的——涉及对合规自动化软件（无论是否带有智能体支持）进行评测基准化。

\subsection{A.3 FinOps（财务运营）}

IT 成本管理这一领域涵盖了多个学科，即 FinOps、IT 财务管理（IT Financial Management，ITFM）、技术业务管理（Technology Business Management，TBM）以及项目组合业务管理（Portfolio Business Management，PBM）。目前，FinOps 领域通常处理云成本 (Storment and Fuller, 2023; Yang et al., 2024b)，这些成本来自计算节点、内存、其他存储、网络等资源，并由某一超大规模云服务商（hyperscaler）产生。ITFM 则涵盖本地基础设施、许可、人工、采购的服务、技术支持等。TBM Council 提供了一套标准的分类法，用于描述成本来源、技术、IT 资源（IT towers）、应用与服务。此外，还存在针对特定行业（如医疗、银行等）对该分类法的扩展。从本质上说，这套分类法提供了一种普遍接受的方式来对 IT 成本及其他指标进行归类与报告。PBM 指的是在组织内部管理一系列项目与计划的实践，以确保其与整体业务战略保持一致，并通过高效配置资源来最大化其集体价值。FinOps 基金会（FinOps Foundation）已表示，未来它将逐步纳入来自 ITFM、TBM 与 PBM 的要素。

目前，对于本文所采用的 FinOps 评测基准定义，尚不存在与之契合的评测基准。然而，多年来，FinOps 基金会 (Foundation, 2025a) 已编制了若干 KPI，可作为构建 FinOps 评测基准用例与场景的基础。当前 FinOps 基金会的 KPI 包括：

\begin{itemize}
\item 用量或支出分摊验证（Usage or Spend Apportionment Validation）
\item 支出的总未预测方差（Total Unpredicted Variance of Spend）
\item 由承诺折扣（commitment discount）覆盖的计算支出百分比（Percent of Compute Spend Covered by Commitment Discounts）
\item 有效节省率百分比（Effective Savings Rate Percentage）
\item 承诺折扣浪费百分比（Percentage of Commitment Discount Waste）
\item 未使用资源百分比（Percent of Unused Resources）
\item 自动扩缩效率率（Auto-scaling Efficiency Rate）
\item 预测准确率（用量、支出）（Forecast Accuracy Rate (Usage, Spend)）
\item 未分配的共享 CSP 云成本百分比（Percentage of Unallocated Shared CSP Cloud Cost）
\item 预算与预测的 CSP 云支出方差百分比（Percentage Variance of Budgeted vs. Forecasted CSP Cloud Spend）
\item 符合标签策略（tagging policy）的 CSP 云成本百分比（Percentage of CSP Cloud Costs that are Tagging Policy Compliant）
\item 位于高频访问存储层（Frequent Access Tier）的存储百分比（Percent Storage on Frequent Access Tier）
\item 遗留资源百分比（Percentage of Legacy Resource）
\end{itemize}

随着云计算的出现，学术界与工业界的研究社区也一直积极探索在平衡多重目标的同时优化成本的方法。FinOps 领域的近期工作聚焦于应用机器学习与数学优化技术 (Qiao et al., 2021; Yang et al., 2024b)，以更好地服务客户的云基础设施需求，同时为其提供如何优化整体云支出的洞见与建议。(Fangkai et al., 2023) 针对在存在多种供给选项（例如比按需虚拟机更便宜但可用性较低的竞价型虚拟机，spot VM）情况下，帮助客户在成本与资源可用性之间权衡取舍的问题。他们提出了一个框架，通过识别按需虚拟机与竞价型虚拟机的最优组合，使用受约束的强化学习（constrained reinforcement learning）来同时优化成本与可用性。诸如 (Diao et al., 2024; Feng et al., 2023; Quattrocchi et al., 2024) 之类的论文提出了用于针对服务等级目标扩缩云资源的预测算法，为更广泛的、由 FinOps 驱动的成本优化领域作出贡献。(Osypanka and Nawrocki, 2020) 使用异常检测、机器学习与粒子群优化来实现成本最优的云资源配置。(Liu et al., 2023a) 从用户视角出发，分析了使用云存储进行成本优化的过程，以探究其中的机会、动因与挑战。(Nodari et al., 2016) 聚焦于寻找按需实例与预留实例的最优组合，从而在满足需求的同时使成本最小化。他们将这一优化问题建模为一个随机库存控制问题（stochastic inventory control problem）。

(Yehoshua et al., 2023) 提出了一个可扩展的成本优化器，它在考虑资源需求与约束以最小化成本的前提下，为公有云或混合云上的工作负载确定最具成本效益的部署策略。在 FinOps 领域，迫切需要超越比较性记分卡（comparative scorecards）与宽泛的分类法，转而面向那些能够测试自动化智能体优化 IT 投资、减少资源浪费能力的具体用例。据我们所知，目前尚不存在任何针对预测、异常检测或成本优化等用例的评测基准，也没有标准化的方法来评估这些技术（无论是否带有智能体支持）。我们坚信，ITBench 将凝聚研究与开发社区的力量，借助 AI 与优化的力量来攻克真实世界中的难题。

\subsection{B.3 排行榜（Leaderboard）}

对 IT 自动化任务进行有效的评测基准化——尤其是在为某一组织的特定需求选择 LLM 时——要求对智能体性能进行一致的追踪与比较。排行榜（Leaderboard）通过提供一套预定义、可扩展的性能指标来满足这一需求，这些指标能够针对评估标准给出清晰的智能体性能洞见。

排行榜同时支持 API 与 UI 两种接口，从而实现流畅的评测基准化工作流。用户必须通过排行榜的 UI 或 API 注册其智能体端点（endpoint）。随后，智能体即可向排行榜查询，以检索并部署基准场景，然后报告其运行状态。如前所述，场景既可以由 ITBench 在其托管环境中自动部署，也可以在排行榜之外、即用户自己托管的环境中手动部署——在后一种情况下，智能体与环境仍可利用同一个排行榜 API 端点来发布状态更新。

智能体在被智能体提交者注册之后，其端到端的基准化工作流如图 5 所示，并总结如下：

\begin{enumerate}
\item 新的基准作业被存入基准队列（Benchmark Queue）以待处理。
\item 基准运行器（Benchmark Runner）从基准队列中为某一特定智能体获取一个基准场景。
\item 基准运行器依据基准场景规约来提供（provision）环境。场景的环境是执行某一具体 IT 任务所需的系统集合。智能体与该环境交互，以求解给定的 IT 自动化任务（并有可能修改该环境）。基准评测会根据智能体是否在给定环境中成功完成任务来衡量其性能。该环境可以是，例如，一个运行着目标应用的 Kubernetes 集群，或一台具有待校验的特定配置的 RHEL 9 主机。该环境处于智能体的直接控制之下，因此可能遭受破坏性操作（例如在故障性能场景中），从而起到某种“试验场（playground）”的作用。
\item 对于基准运行中包含的每个场景，基准运行器与智能体执行以下步骤：
\begin{enumerate}
\item[(a)] 智能体持续轮询 \textbf{get\_manifest} API，以监测何时有新的清单（manifest）进入 \textbf{Ready}（就绪）状态。
\item[(b)] 基准运行器通过执行 \textbf{deploy\_scenario} 函数来部署该场景的环境。每个环境使用 \textbf{post\_bstatus} API 向智能体 API 服务器（Agent API Server）报告其状态。
\item[(c)] 基准运行器通过智能体 API 服务器的 \textbf{get\_bstatus} API 监测环境状态。一旦状态变为 \textbf{Deployed}（已部署），它便通过执行 \textbf{inject\_fault} 函数向环境中注入故障。
\item[(d)] 基准运行器继续使用 \textbf{get\_bstatus} API 监测环境状态。一旦状态到达 \textbf{FaultInjected}（已注入故障），它便在基准数据库（Benchmark DB）中将该清单的状态更新为 \textbf{Ready}，并在清单中写入诸如基准 ID、场景 ID、集群凭据以及 URL 等关键细节。这样，智能体便可访问并检索该清单，以便与环境进行交互。
\item[(e)] 一旦清单状态变为 \textbf{Ready}，智能体便将其检索回来。该清单包含启动智能体所需的 URL 与凭据。在启动之前，智能体会调用智能体 API 服务器的 \textbf{post\_status} API，将其状态报告为 \textbf{STARTED}（已启动）。
\item[(f)] 在智能体完成其执行后，会再次调用 \textbf{post\_status} API，将其完成状态报告为 \textbf{FINISH}（已完成）。
\item[(g)] 基准运行器启动评测，并执行 \textbf{delete\_scenario} 函数。
\end{enumerate}
\item 一旦基准中所有场景的评测结果均已就绪，基准运行器便将它们汇总，并将结果发布到排行榜。
\end{enumerate}

\subsection{C.1 背景}

随着现代 IT 系统与基础设施在规模和复杂度上空前增长，故障已成为常态而非例外 (Patterson et al., 2002; Gunawi et al., 2016; Kendrick, 2012; Di Martino et al., 2014; Veeraraghavan et al., 2018; Liu et al., 2019; Ghosh et al., 2022)。首先，\textit{硬件故障}（hardware failures）在大规模 IT 基础设施中频繁发生。例如，Google 的一个新集群每年要经历约一千次单机故障和数千次磁盘故障 (Dean, 2009)。其中许多故障还会进一步引发相关联的级联故障 (Ford et al., 2010)。诸如计算单元中的静默数据损坏 (Hochschild et al., 2021) 和慢速故障（fail-slow）存储 (Gunawi et al., 2018) 等新型硬件故障模式，进一步加大了检测与缓解（mitigation）的难度。在地理分布式的超大规模（hyperscalar）基础设施中，数据中心级别的灾难已不再罕见 (Veeraraghavan et al., 2018)。

此外，软件变更的高速度也带来了挑战：由代码缺陷 (Gunawi et al., 2014) 和错误配置 (Xu et al., 2013) 引发的\textit{软件故障}（software failures），近年来也已成为导致 IT 系统与服务中断的主要原因，其数量明显超过硬件故障 (Maurer, 2015; Barroso et al., 2018)。例如，IT 系统每天会进行数百到数千次配置变更，这会引入错误配置并触发潜伏缺陷 (Sun et al., 2020; Tang et al., 2015)。微服务和无服务器计算 (Jonas et al., 2019) 等近期的软件架构趋势，通过放大系统复杂度、引入复杂交互所带来的动态性 (Tang et al., 2023) 以及涌现行为 (Huang et al., 2022)，进一步加剧了 IT 可靠性方面的挑战。

站点可靠性工程（SRE，Site Reliability Engineering）的目标是在存在上述各类故障的情况下，实现 IT 系统的高可用性与可服务性 (Murphy et al., 2024)。

SRE 的核心工作是故障管理\footnote{我们采用经典的“故障—错误—失效”模型（Fault-Error-Failure model）(Avizienis et al., 2004)，其中\textit{故障}（fault）是根因（root cause），例如软件缺陷、硬件故障或错误配置。故障可能产生被称为\textit{错误}（errors）的异常行为。然而，部分错误是瞬态的，对系统层面没有影响。只有那些传播并变得可观测的错误，才会表现为\textit{失效}（failures），例如崩溃、挂起、错误结果或功能不完整等。}——即检测、诊断（diagnosis）并缓解生产系统中的故障，以预防生产\textit{事件}（incidents，指造成用户感知影响的失效），或在事件\textit{告警}（alerts）被触发时最小化其影响与损害。具体而言：

\begin{itemize}
\item \textbf{检测（Detection）。} SRE 必须及时通过日志、链路追踪（traces）及其他遥测数据检测生产故障；检测故障是预防事件、或至少最小化其爆炸半径与影响的第一步。
\item \textbf{诊断（Diagnosis）。} SRE 必须分析所检测到故障的根因并定位故障，即确定出故障的组件以及触发该故障的条件。
\item \textbf{缓解（Mitigation）。} SRE 必须缓解故障以防止其传播为更大的失效或事件。缓解通常遵循一份解决方案计划（resolution plan），其中列出了一系列将系统恢复至期望状态的操作 (Chen et al., 2024b)。
\end{itemize}

ITBench 目前聚焦于诊断与缓解任务，并计划纳入更多任务，如事件检测、类似故障/事件的预防（例如通过回归测试）。检测可借助可观测性（observability）工具原生提供的、基于黄金信号（golden-signal）的告警来简化。不过，在事件风暴（event storm）期间难度会加剧，这要求 SRE 通过抑制误报来区分可处置的告警，并对其中需要立即关注的告警进行优先级排序——这是事件解决中每天都要面对的难题。这两类任务都已被纳入 ITBench，方法是在特定场景中注入多个故障，造成告警泛滥；智能体随后必须判断应优先处理哪些告警以及处理顺序。

\textbf{SRE 自动化的迫切需求。} 目前，SRE 在很大程度上仍是一种基于人工的实践——SRE 工程师每天身处检测、诊断和缓解故障与事件的第一线 (Beyer et al., 2018; Murphy et al., 2024)。然而，IT 系统的规模不断增长，其需求已超出基于人工的实践所能可靠、持续、高效管理的范围，而且人力资源成本以及人类推理的极限，已成为故障与事件解决的瓶颈。如今，IT 系统的 SRE 已成为任何云与软件公司的主要总拥有成本（TCO，Total Cost of Ownership）(Boulton; IDC, 2024)。\textit{因此，SRE 自动化不再是一项可选的增强，而是一种运营上的必需。}

事实上，当今的 IT 系统已越来越多地由实现劳动密集型、基于人工操作自动化的运营程序来管理，这类自动化被称为 \textit{IT 自动化}（IT automation）。例如，Kubernetes (Burns et al., 2016)、Twine (Tang et al., 2020) 和 ECS (Melissaris et al., 2022) 等现代云管理平台都实现了\textit{算子}（operator）程序，以自动化软件升级、配置、自动扩缩容等各类操作。然而，迄今为止，由于故障管理中存在的根本性挑战，SRE 尚未成为 IT 自动化的常见组成部分。

\subsection{C.3 ITBench 架构}

ITBench 使用开源技术，在 Kubernetes 平台上创建完全可重复、可复现的事件（场景），如图~\ref{fig:itb7} 所示。

\textbf{编排（Orchestration）。} 核心工作流涉及 SRE-Agent\footnote{此后我们将处理 SRE 任务的智能体称为 SRE-Agent。} 与 ITBench 各组件之间的一系列交互。首先，SRE-Agent（\textcircled{1}）通过发送 \texttt{enroll} 命令在评测排行榜中注册，这会促使 ITBench 创建一个会话（session）（\textcircled{2}）并提供必要的凭据与细节（例如 Kubernetes 访问权限、时间限制）。准备就绪后，智能体发送 \texttt{ready} 信号（\textcircled{3}），触发场景执行器（scenario executor）从场景数据库中安装选定的场景。该规范用于搭建环境并注入故障，包括可观测性工具的安装（\textcircled{4}）。在活动阶段（\textcircled{5}），智能体使用 \textit{NL2Alerts}、\textit{NL2Logs}、\textit{NL2Metrics} 和 \textit{NL2Traces} 等工具与环境交互以完成任务。任务完成或时间到期后，SRE-Agent 发送 \texttt{finish} 命令（\textcircled{6}），通知 ITBench 评估所提供的输出并清理环境。场景执行器校验 SRE-Agent 的工作（\textcircled{7}），并将系统恢复到其基线状态（\textcircled{8}）。\textcircled{3}—\textcircled{8} 之间的交互会持续进行，直至场景管理器发送会话结束信号（\textcircled{9}）。

\subsubsection{C.3.1 原则}

遵循引言中提出的 bench 原则，我们的 ITBench 使用开源技术来构建完全可重复、可复现的场景，以模拟真实世界事件。

\begin{itemize}
\item \textbf{模仿 SRE 最佳实践。} ITBench 遵循 SRE 手册 (Murphy et al., 2024) 中列出的准则，例如按应用对黄金信号进行告警、启用监控与可观测性。在我们当前的版本中，检测是采用 (Murphy et al., 2024) 中所述方法开箱即用地提供的。
\item \textbf{模仿真实世界事件。} 我们系统性地考察了来自我方 SaaS 产品的 105 个真实世界事件，以推导出相关的事件模式。尽管我们已将其中若干模式集成到 ITBench 场景中，但并非全部都被纳入，原因在于准确复现这些事件、并镜像出生产级细节特征所固有的复杂性。尽管如此，我们的 ITBench 将通过持续纳入更多事件模式而不断演进。截至撰写本文时，ITBench 支持 24\% 的 Easy、24\% 的 Medium 和 52\% 的 Hard 事件，如图~\ref{fig:itb9} 所示。
\item \textbf{提供可观测性。} 在真实世界场景中，SRE 会结合命令行访问，使用可观测性工具来监控系统。这些工具提供多种数据模态（modality），如链路追踪、日志、指标与事件——并支持告警，以实现高效的异常检测、趋势分析与自动化故障排查。ITBench 默认使用 Grafana (gra)，但也可支持其他工具，包括 IBM Instana (ins)、Dynatrace (dyn) 和 Datadog (dat)。
\item \textbf{数据模态可变性。} 根据系统的关键程度与预算，部分数据模态可能缺失；例如，只有约 20\% 的应用启用了链路追踪，这使事件诊断更加复杂。ITBench 允许按需灵活控制以启用、禁用或部分启用各数据模态。
\item \textbf{管理可扩展性。} 场景超参数由 (i) 环境规范和 (ii) 场景规范两部分构成。环境规范允许进行应用选择及其相关的基础设施选择（例如副本数）以及数据脱敏参数设置。场景规范允许选择超参数（例如注入故障所针对的服务名称）。ITBench 使用上述超参数离线创建一个场景数据库。
\item \textbf{确保确定性。} ITBench 确保在使场景在 ITBench 中可用之前，告警是按照场景规范生成的。此外，ITBench 会确保在向智能体发送“READY”状态信号之前，所有断言（assertions）（例如应用正常运行、告警正确触发）都已通过。
\end{itemize}

\subsection{C.4 ITBench 事件特征刻画}

表~\ref{tab:itb12} 概括了 ITBench 中当前可用的场景。除了这 21 个场景之外，ITBench 还能通过参数化关键维度（如目标应用、故障注入的精确位置以及并发故障的数量与类型）轻松产生范围广泛得多的故障模式。例如，如果目标应用是 HotelReservation，则仅 PodFailure 场景的故障就可以应用于其 18 个 pod 中的任意一个，从而有效扩展出另外 18 个场景。通过这种方式，ITBench 可被用于系统性地生成成百上千种变体。在我们的评估中，我们聚焦于具有代表性的场景，同时仍允许用户定制并扩展其测试。

图~\ref{fig:itb8} 展示了我们数据集中所观测到的关键事件特征，包括\textit{故障传播链长度}（fault propagation chain length，图~\ref{fig:itb8}a）、\textit{解决方案计划规模}（resolution plan size，图~\ref{fig:itb8}b）以及所涉及的\textit{独特技术数量}（number of distinct technologies，图~\ref{fig:itb8}c）。直观上，随着故障传播链长度的增加，事件变得更难诊断。类似地，更长的解决方案计划意味着恢复服务健康需要多个步骤与干预。涉及各类技术会引入额外的复杂度，原因在于工具、数据源与故障模式的多样性。

由于\textit{故障传播长度}、\textit{解决方案计划规模}和\textit{技术异构性}（technology heterogeneity）都会影响事件解决的难度，我们将整体任务复杂度定义为三者的几何平均（geometric mean）。式~(6) 刻画了这一关系：

\begin{equation}
\text{Complexity} = \sqrt[3]{\text{(propagation path length} \times \text{\# resolution steps} \times \text{\# technologies)}}
\tag{6}
\end{equation}

\subsection{C.5 SRE-Agent}

如第 3.2 节所述，智能体（agent）与目标环境交互、收集可观测性（observability）数据，并执行动作以达成其目标。对于 SRE（站点可靠性工程，Site Reliability Engineering）而言，目标是诊断（diagnosis）并缓解（mitigation）故障事件。下面我们描述 SRE-Agent 所收集的可观测性数据，以及我们基于大语言模型（LM）的多智能体系统实现。

\subsubsection{C.5.1 可观测性数据}

\begin{figure}[H]
\centering
\includegraphics[width=0.78\linewidth]{cand_p27_k00.pdf}
\caption{用于 SRE 任务的多模态数据。}
\label{fig:itb-fig10}
\end{figure}

如图 10 所示，SRE 任务涉及分析多模态可观测性数据：日志（logs）、链路追踪（traces）和指标（metrics）。

\textbf{日志（Logs）。} 日志是半结构化的文本记录，用于捕获硬件与软件事件。它们通常按严重级别归类，如 INFO、WARN 和 ERROR，以反映系统的运行状态以及问题的严重程度。

\textbf{链路追踪（Traces）。} 请求链路追踪描述用户请求在分布式系统中穿越各服务实例时的执行流程。它们提供服务调用的分层表示，其中每个操作被称为一个跨度（span）。一个跨度记录单次服务调用的信息，如其起始时间、持续时长以及相关元数据（包括标签和日志）。多个跨度相互关联形成一条 trace，从而捕获请求的完整执行路径。此外，程序执行链路追踪还可捕获程序崩溃信息，为开发者在调试时提供宝贵线索。

\textbf{指标（Metrics）。} 指标提供监测系统性能与用户感知指标的时间序列数据，如延迟、错误率和资源利用率。

\subsubsection{C.5.2 SRE-Agent 架构与实现}

SRE-Agent 架构由两个基于大语言模型的智能体构成：一个诊断智能体（Diagnosis Agent）和一个修复智能体（Resolution Agent），如图 11 所示。我们首先定义实现中所用的以下基本组件：

\begin{itemize}
\item \emph{智能体（Agent）。} 智能体是一个自主或半自主的软件程序，它利用大语言模型进行规划、决策、与目标环境交互并执行动作以达成目标。
\item \emph{任务（Task）。} 任务是智能体在其执行终止前必须达成的特定目标。在我们的实现中，任务是一个复杂的多步骤流程（例如诊断某次故障事件的成因）。任务还关联有智能体可用于达成目标的工具。
\item \emph{工具（Tool）。} 工具是智能体可调用以执行某一具体子任务的函数或 API 调用，例如与目标环境交互以收集可观测性数据。
\end{itemize}

下面我们描述上述各组件的具体实现。

\textbf{工具（Tools）。} 表 14 列出了 SRE-Agent 可用的全部工具。我们所有的工具同样基于大语言模型：以一段来自智能体、指示其执行所需子任务的话语（utterance）作为提示输入大语言模型。根据工具是生成自然语言（如 Mitigation）还是生成函数调用（如 NL2Kubectl），可将工具分为两类。此外，为提升工具的准确性与可用性，我们为工具配备了以下特性。

\item \emph{反思（Reflection）。} 为实现对错误大语言模型响应的自动纠正，我们借助 linter（代码检查器）提供外部反馈（Pan et al., 2023；Huang et al., 2023）。具体而言，针对生成函数调用的工具，我们开发了 linter 来校验输出的语法与语义。如果 linter 发现生成的函数调用存在问题，便会将 linter 的反馈连同问题一并重新提示给大语言模型，使其尝试修正。类似地，如果生成的函数调用通过了检查但在执行时引发错误，则将错误消息与失败的函数调用一并重新提示给大语言模型，以获取修正后的函数调用。

\end{itemize}

\textbf{任务（Tasks）。} 我们定义了 SRE-Agent 需完成的以下两项任务。每项任务都包含对其完成过程的描述以及完成后期望的输出。每项任务还关联有工具，智能体可用其执行子任务、收集信息或与环境交互。

\begin{itemize}
\item \emph{诊断任务（Diagnosis Task）。} 对于诊断，目标是识别完整的故障传播链（fault propagation chain，FPC），即故障传播链分析；并在该链条中识别问题的确切成因，即故障定位（fault localization，FL）。
\item \emph{缓解任务（Mitigation Task）。} 对于缓解，目标是提供自然语言形式的缓解方案，并执行这些方案以成功清除触发的告警。缓解方案可提升智能体的可解释性，并帮助 SRE 理解智能体为何执行了某些命令。
\end{itemize}

\textbf{智能体（Agents）。} 总体而言，SRE-Agent 由两个智能体构成，即诊断智能体与缓解智能体。每个智能体被指派其必须完成的任务。一般来说，多智能体系统可以是\emph{分层式（hierarchical）}或\emph{顺序式（sequential）}的。顺序执行使任务以固定的线性顺序完成。在分层执行中，由一个“管理者（manager）”智能体确定任务的执行顺序并与其他智能体协调。我们采用顺序执行，因为它非常适合 SRE 用例——故障事件必须先经诊断才能被修复。尽管任务执行顺序是固定的，但每项任务内部的子任务或步骤可由智能体自身决定按任意顺序完成。下面我们描述两个智能体的整体工作流程。

\centering
\includegraphics[width=0.85\linewidth]{cand_p29_k00.pdf}
\caption{SRE-Agent 架构。}
\label{fig:itb-fig11}
\end{figure}

\begin{itemize}
\item \emph{诊断智能体（Diagnosis Agent）。} 首先，诊断智能体使用 NL2Alerts 工具检索环境中的活动告警。随后，智能体灵活而迭代地使用可观测性工具，从告警中提及的受影响实体以及与该实体关联的实体处收集 trace、日志和指标。它还可能使用 NL2Kubectl 命令来调查环境。一旦智能体确定已掌握足够信息可给出诊断，便会生成一份 JSON 格式的结构化诊断报告，将其发现记录其中以便于评估。报告生成后，缓解智能体接手。
\item \emph{缓解智能体（Mitigation Agent）。} 缓解智能体读取诊断报告以制定缓解方案，随后利用可用工具实施该方案。这涉及使用 NL2Kubectl 命令。为确保所执行的命令已缓解故障事件，它还可使用 NL2Alerts 工具检查环境中的告警是否已被清除。此外，由于在实时环境的波动下告警有时可能暂时显现为已清除，智能体可使用 Wait 工具检查告警在一段时间后是否\emph{仍然}保持清除状态。最后，执行完成后，智能体生成一份 JSON，说明其所采取的缓解步骤。
\end{itemize}

\subsubsection{C.6.2 评测指标}

我们在两项主要任务上评测每个基于大语言模型的智能体：(i) 诊断和 (ii) 缓解。

\textbf{诊断（Diagnosis）。} 智能体的诊断能力根据其提供准确\emph{故障定位}与\emph{故障传播链}的能力来评测。故障定位使 SRE 能够识别\emph{导致}问题的确切资源，而故障传播链使 SRE 能够理解故障如何在应用栈中级联传播并影响应用。故障传播链还可进一步用于其他重要任务，如爆炸半径（blast radius）分析。

\begin{itemize}
\item \emph{故障定位}性能使用 pass@1 和归一化拓扑感知匹配（Normalized Topology-Aware Match，NTAM）来度量。
\item \emph{故障传播链}使用 NTAM 评估。此外，我们追踪平均诊断时间（Mean Time to Diagnosis，MTTD）以衡量整体诊断效率。
\end{itemize}

\textbf{缓解（Mitigation）。} 对于缓解，我们评测智能体修复故障事件（即清除告警）的有效程度。

\begin{itemize}
\item 成功率使用 pass@1 量化。
\item 效率通过平均修复时间（Mean Time to Resolution，MTTR）来刻画。
\end{itemize}

\subsubsection{C.6.3 指标定义}

\textbf{pass@1。} 我们使用 pass@1 指标（Chen et al., 2021）来评测故障定位与缓解，其定义如下：

\begin{equation}
\mathrm{pass@}k := \mathbb{E}_{\mathrm{Scenarios}}\left[ 1 - \frac{\binom{n-c}{k}}{\binom{n}{k}} \right]
\end{equation}

它是所有场景上 $k=1$ 试验正确性的无偏估计量。对于\emph{故障定位}，正确性指预测的根因（root cause）是否与真实根因完全匹配。对于\emph{缓解}，正确性指告警是否被清除。

\textbf{归一化拓扑感知匹配（Normalized Topology-aware Matching）。} 现有用于评估\emph{故障传播链}与\emph{故障定位}的方法侧重于与真实标签的精确匹配（Ahmed et al., 2023b；Zhu et al., 2024；Chen et al., 2024c），从而忽略了拓扑结构以及对传播链更细粒度的分析。例如，当预测的传播链或根因与真实标签不完全匹配时，现有方法无法有效区分智能体与模型，因为它们未能度量预测与实际故障的接近程度。因此，我们提出一种新指标——归一化拓扑感知匹配（NTAM），通过拓扑感知的距离计算来度量智能体相对于真实标签的性能。

NTAM 需要一张拓扑图，其中节点为系统的实体，边表示它们之间不同类型的连接（例如 Deployment 拥有 ReplicaSet）。给定这样一张拓扑图，便可用其评估故障传播链中的实体集合，以及用于故障定位的根因实体集合。NTAM 受拓扑感知的距离度量以及信息检索概念（如 BM25，Fang et al., 2011）启发，后者会对区分度较低的特征进行降权。它是一个灵活、通用的函数，具有可配置组件，可对预测输出质量进行细粒度评估。

具体而言，它由以下主要组件构成：

\begin{itemize}
\item \emph{基于拓扑的距离评分}函数，同时考虑边类型与子树规模，对更接近真实标签的预测实体给予奖励。此外，连接较少（子树较小）的节点获得更高分数，因为它们对故障定位更具区分度。
\item \emph{节点重要性因子}，基于真实标签实体在传播链中的位置而定。这体现了如下直觉：正确预测真实根因实体所应获得的奖励，应高于在链条上预测对了另一个实体。
\item \emph{长度失配的惩罚项}，针对预测实体与真实实体之间的数量差异。这是为了确保预测实体过多或过少时得分更低。
\end{itemize}

所有组件都有相应的超参数，可调以调整它们对总分的贡献。最终得分被归一化到 0 与 1 之间，其中 1 表示完全匹配。对于故障定位，不评估全部实体集合，而是仅考虑真实与预测的根因实体。

\textbf{平均诊断时间（Mean Time to Diagnosis）。} 对于智能体成功完成诊断（即找到根因实体）的场景，我们计算 MTTD，用以度量智能体执行诊断的快慢（以秒为单位）。否则，MTTD 设为无穷大。

\textbf{平均修复时间（Mean Time to Repair）。} 类似地，对于缓解，我们识别出智能体成功执行自动化动作以修复故障（即告警被清除）的场景。对于这些场景，我们计算 MTTR（以秒为单位），用以度量智能体执行缓解的快慢。否则，MTTR 设为无穷大。

\textbf{诊断与缓解之间的相互依赖。} 有趣的是，诊断与缓解常被直觉地认为相互依赖，即准确的诊断是有效缓解的前提条件。然而，\emph{我们的发现揭示：在某些场景中，智能体尽管错误识别了根因，却仍能成功缓解故障事件}。例如，在场景 15（图 12）中，使用 llama-3.3-70b 模型的智能体将根因错误识别为服务中的“内存限制（memory limit）”，而真实根因是 HTTP 请求损坏故障；然而它仍设法通过扩容 email 服务的 pod 数量使其恢复功能，从而实际上绕过了对真实 HTTP 故障的处理。此类案例说明，诸如重启服务或扩容副本之类的通用缓解动作，有时即使在缺乏完全准确诊断的情况下也能修复系统症状。我们在真实世界的 SRE 故障事件分析中也观察到类似行为：尽管根因仍未确定，SRE 仍能够缓解故障事件。

相反地，某些场景揭示了相反的问题：场景 13 虽被标注为“简单”，却无法被任何受测智能体修复，即便它们在诊断根因上取得了高分。值得注意的是，gpt-4o 取得了约 80\% 的 pass@1，并在 FPC 与 RC（NTAM）指标上均超过 0.7。这意味着，\emph{尽管智能体无法完全解决某些问题，它仍能提供近乎准确的诊断洞察，从而潜在地辅助人类操作者进行调试}。

某些场景产生了更差的结果。例如，在场景 19（图 13）中，智能体未能识别根因，也无法修复系统，最终给出的缓解方案完全无效。

\textbf{运行间的不一致性。} 另一个关键观察是智能体在重复运行间的不一致性。例如，llama-3.1-8b-instruct 和 mistral-large-2 在场景 11 中偶尔仅在 10 次运行中成功 1 次。尽管 gpt-4o 能可靠地修复场景 8，但其运行时间可在 100 至 800 秒之间波动。\emph{这些不一致性源于实时遥测的波动，其中微小的变化（例如某次运行报告 CPU 利用率为 58\%、另一次为 71\%）会影响大语言模型的输出，从而导致诊断与缓解结果各异。}

\textbf{链路追踪对准确率的影响。} 许多基准提供原始遥测数据，而 ITBench 的一个关键差异化特征是其告警驱动的工作流，这与 SRE 通过由所采集遥测数据触发的黄金信号告警（golden-signal alerts）获知故障的方式相吻合。为进一步评估不同遥测来源的重要性，\emph{ITBench 还支持自动化的遥测数据掩蔽（telemetry data masking）}。如表 19 与表 20 所示，gpt-4o 的诊断 pass@1 从 18.10\%（有 trace）降至 9.52\%（无 trace），其缓解 pass@1 骤降至 2.86\%。类似地，llama-3.3-70B 的诊断率从 5.24\% 降至 0.95\%。事实上，一旦 trace 数据被掩蔽，仅有三个场景能被 gpt-4o 成功解决。以场景 13（简单难度）为例，智能体在所有运行中均能达到 80\% 的诊断率；然而当掩蔽 trace 时，该比率降至 0。需要注意的是，这些遥测掩蔽与智能体评估步骤都已集成进 ITBench 的自动化流水线。智能体可在不同的可观测性配置下接受评估。

\textbf{灵活性与可扩展性。} 除场景设计外，\emph{ITBench 以灵活性与可扩展性为指导原则进行设计，既支持在现有任务内新增场景，也支持新增任务（如资源管理）}。我们已集成四种不同的应用，包括微服务应用（OpenTelemetry-Demo 以及来自 DeathStarBench 的 Hotel-Reservation）和非微服务应用（TiDB 应用与 Elasticsearch 应用）。ITBench 通过简单添加一个 Ansible playbook，便可直观地纳入自定义应用。外部协作者将一个应用接入 ITBench 大约耗时四个人工小时。此外，ITBench 在多个系统层（如应用层、虚拟化层）注入真实故障，确保对智能体性能进行覆盖广泛故障类型的全面评测。

\section{附录 D：CISO（合规与安全运营）与合规评估智能体评测}

\subsection{D.1 背景}

技术的进步正在不断加剧应用与基础设施的复杂度。因此，那些依赖专职安全与合规团队、根据漏洞对组织构成的威胁来识别生产系统漏洞并加以缓解的传统做法，已经无法奏效。现代组织依赖一种开发、安全与运营一体化（DevSecOps）的实践，即在部署前对应用安全进行验证：在软件开发阶段就植入安全与监管管控；随后在部署后由运行时检查接力完成校验。在这些由不同团队负责、且其中部分团队网络安全知识有限的多层级安全与合规检查之下，手动的合规流程不仅对技术性安全管控而言已不再可行，而且这些流程的自动化也需要以前所未有的速度和规模加以推进，以跟上产品上市的节奏。

整个流程始于 CISO（合规与安全运营，Chief Information Security Officer）、安全管理员或监管方建立并撰写相关的合规建议正文，通常以自然语言、针对特定的关键任务环境给出。随后他们依靠开发团队或安全专员在这些环境中收集状态作为证据，并出于取得运营授权（authorization to operate）或其他认证的目的，对照建议对其进行校验。自动化所追求的收益在于自动化\textbf{证据收集（evidence collection）}及其\textbf{校验（validation）}，从而实现可扩展性——既能处理复杂环境，也能支持频繁扫描（每日或按需，在每次系统修复或更新后进行）以完成状态度量与报告。校验自动化工具的例子包括策略引擎，如面向 Kubernetes 的 Kyverno（Int, c）、OPA Gatekeeper（Int, b），面向 PaaS 的 Ansible（Int, a），或面向云的云安全态势管理（CSPM）解决方案。图~\ref{fig:itbench-19} 展示了这种二分对立：左侧为合规撰写（compliance authoring），右侧为通过策略脚本及其多样化的编程语言进行的合规校验（compliance validation）。

把自然语言建议自动翻译为策略脚本，要求在通常分属不同业务单元的各个领域与专家之间，达成前所未有的信任与同步水平。此外，它还要求通常专注于法律与知识产权（IP）事务的合规团队具备前所未有的技术知识。

AI 智能体（agent）日益普及，以及人们预期其将能够处理复杂任务，使得对管理 IT 系统的 AI 智能体的需求随之上升（John, 2024；Miguel Carreon, 2024）。鉴于合规任务的复杂性，本研究面临的一大障碍是：要建立系统化的方法，在 AI 智能体投入生产部署之前评估其有效性。因此，迫切需要开发基于真实 IT 任务及其对应环境来评估 AI 智能体的方法。

我们在开源（CIS, a）的 git 代码库文档中详细介绍了 CISO 合规评估智能体的部署与执行。下文将通过一套定义良好、含有真实场景与环境的方法论，展示我们的 CISO 智能体评测性能。下文随后给出一组场景样例及其环境对应的可执行自动化包，这些也在开源（CIS, b）中提供。

\subsection{D.2 真实世界评测}

我们的 CISO 合规评估智能体（CAA，Compliance Assessment Agent）及其对应的评测基准（bench）将合规即代码（compliance as code）领域的最新技术结合在一起，借助生成式 AI（Gen AI）的代码生成能力，实现对监管管控及其态势评估的程序化表达。我们的智能体旨在赋能合规团队，加速新监管程序的采纳与运营：根据以自然语言描述的合规要求，自动生成用于证据收集及对照要求进行态势校验的代码。我们的评测实验覆盖端到端的智能体式（agentic）工作流，从在评测场景中发现策略技术，到将评估策略生成为代码，再到其实时的 git PR 管理、部署、执行与态势生成。最后，结果在我们的 ITBench 解决方案排行榜（leaderboard）上被评估、评级并报告。

\subsubsection{D.2.1 术语与记号}

我们定义以下关键术语，下文将用它们来描述智能体框架与评测方法论的主要方面：

\begin{enumerate}
\item \textit{智能体（Agent）：}智能体是一种由 AI 驱动的软件，它代表某一角色（persona）自主行动以解决给定任务。我们按照反映 IT 运营角色的智能体类型（Agent Type）对智能体进行分组，例如 CISO、FinOps 或 SRE 类型。
\item \textit{任务（Task）：}任务是与某一角色职责相对应的具体工作，智能体旨在将其自动化。CISO 的典型任务包括收集证据和评估合规管控态势。
\item \textit{场景（Scenario）：}场景是某一任务在给定环境中的真实实例。以 CISO 为例，\textit{每一项}在 OPA 上实例化的 Kubernetes CIS 基准要求都是一个独立的场景。场景可以分组归入类别（class）。
\item \textit{场景类（Scenario Class）：}场景类是一类被归并在一起的真实场景，期望它们对应同一角色表现出相同的行为与结果。场景类的例子包括 OPA 引擎上的 Kubernetes CIS 基准\textit{集合}、Ansible 引擎上的 RHEL9 CIS 基准\textit{集合}，或 Kyverno 引擎上的 Kubernetes CIS 基准更新\textit{集合}。
\item \textit{场景环境（Scenario Environment）：}场景环境是场景中指定部署设置的部分。
\item \textit{环境状态（Environment State）：}一个环境具有可数的状态集合，我们认为这些状态标记了特定时刻的某一特定条件。状态的例子包括：环境初始部署状态、注入故障或不合规配置后的环境失败状态，以及缓解后修复或合规的环境状态。
\item \textit{目标（Goal）：}目标是环境所期望达到的状态，即目标状态（goal state）。
\end{enumerate}

智能体的任务是以最高效的方式，将环境从其初始状态转移到目标状态。智能体可支配的手段包括环境动作（environment actions），即对系统状态进行观测的请求或试图影响系统状态的执动（actuation）尝试。智能体迈向把环境推入目标状态的第一步，是对若干观测动作的结果进行推理，以确定下一个最优步骤。带着对"最优步骤究竟为何"的有力假设，智能体力求寻找并执行策略，以最高效的方式趋向目标状态。

我们的 CISO 合规评估任务与预定义场景相耦合，以便以标准化方式评估智能体交付自动化的有效性。下一节将先详述这些预定义的 CISO 任务，然后再详述我们的 ITBench 与 CISO 场景。

\subsubsection{D.2.2 CISO 合规评估任务}

合规评估任务包含各种活动，旨在将系统的实际状态与策略中以英文描述的期望状态进行比较。基于这一比较，系统给出相对于策略的"通过（pass）"或"失败（fail）"态势。

\begin{itemize}
\item \textbf{识别证据收集器（Identify Evidence Collector，IEC）：}获取证据需要选择适合目标系统特征的收集机制。例如，关于 Kubernetes 集群中某应用状态的证据，需要访问 Kubernetes API，通常借助诸如 \texttt{kubectl} 命令之类的工具。对于主机配置证据，则适合使用诸如 Ansible Playbook 之类的工具。该 IEC 任务及其关联智能体或工具，会识别环境中所用的收集器，以便生成与其对应语言和接口相匹配的脚本。
\item \textbf{识别策略评估工具（Identify Policy Assessment Tool，IPA）：}对照策略评估证据，需要选择合适的策略引擎。对于通用场景，业界使用开源策略引擎 Open Policy Agent（OPA）（Int, d）及其特定的程序化语言 Rego。此外，对于 Kubernetes 特有的配置，Kyverno 策略与 OPA 一同被广泛使用。该 IPA 任务及其关联智能体或工具，会针对手头策略对应的场景识别合适的策略引擎，以便按照该策略引擎的程序化语言和接口生成策略代码。
\item \textbf{收集证据（Collect Evidence，CE）：}该 CE 任务及其关联智能体或工具负责实际的证据收集，包括代码生成、证据收集代码的管理、部署与执行，以便从环境中获取实际证据。要以可靠且可扩展的方式实现这一点，代码的正确放置与执行是必要的。
\item \textbf{扫描评估态势（Scan Assessment Posture，SAP）：}该 SAP 任务及其关联智能体或工具负责生成态势——即证据是否满足场景的 CIS 基准要求。它包括校验代码的生成、代码在策略引擎上的管理、部署与执行，以评估证据并产出合规态势。
\end{itemize}

表~2 总结了我们 ITBench 中最初支持的 CISO 任务，即 CE 与 SAP。其余任务将在后续版本中覆盖。这些任务在 ITBench 中对照预定义的标准场景执行，并与每个场景环境规约下存储在 ITBench 中的真实期望评估态势"pass"/"fail"进行比较。

ITBench 使用开源技术，为 CISO 任务在 Kubernetes 集群上创建可重复、可复现的场景与环境，如图~\ref{fig:itbench-20} 所示。

\subsubsection{D.3.1 原则}

遵循引言中指出的评测基准原则，我们的 ITBench 使用开源技术构建完全可重复、可复现的场景，以模拟真实世界的事件。

\begin{itemize}
\item \textbf{模仿 CISO 最佳实践。}ITBench 遵循美国国家标准与技术研究院（NIST）网络安全框架（CSF）（NIS）为 CISO 与安全团队改进组织网络安全所列出的指引，具体为：（1）\textit{识别（Identify）}需要保护的关键数据、系统、资产与能力；（2）通过限制事件影响的安全措施进行\textit{保护（Protect）}；（3）制定策略以\textit{检测（detect）}不合规，并配以清晰的程序与工具；（4）通过快速消除威胁并缓解损害的计划进行\textit{响应（Respond）}；（5）设计\textit{恢复（recovery）}策略以支持及时恢复正常运营。我们的首个 ITBench 版本覆盖 CISO CSF 的前三项活动，方法是评估以下合规任务：（1）从选定环境中的系统识别并收集证据；（2）实现场景中建议的策略；（3）评估策略态势以检测进入不合规的失败。其余两项 CISO CSF 活动——\textit{响应}与\textit{恢复}——将成为未来某个 ITBench 版本的主题，该版本将支持智能体协作，即借助 SRE 智能体进行缓解。
\item \textbf{模仿 CISO 真实世界问题。}我们在 ITBench 中研究并使用了真实世界的 Cloud Internet Security（CIS）基准（CIS, 2024），这是一套用于保护 IT 云基础设施的最佳实践集合，被全球公认为云安全标准。我们用这些 CIS 基准创建了不同复杂度级别的 CISO 合规策略场景：25\% 简单（Easy）、50\% 中等（Medium）、25\% 困难（Hard）策略（见图~\ref{fig:itbench-21}）。
\item \textbf{提供可观测性（observability）。}云原生计算基金会（CNCF）近期的 Sandbox 项目（OSCAL-compass, 2024）发布了一套合规即代码 SDK，以支持对美国国家标准与技术研究院（NIST）的机器可读合规即代码（OSCAL, 2024）进行编程化的合规自动化使用。ITBench CISO 自动化利用这一方法论来表达 CIS 基准要求，检测要求创建或更新的事件，并触发证据收集与校验代码的创建或更新。
\item \textbf{确保确定性（Determinism）。}ITBench 强制场景及其环境严格按照规约生成，而每个场景后的环境清理（cleanup）则确保为下一次运行提供干净的初始状态。
\end{itemize}

\subsubsection{D.3.2 ITBench 架构}

评测场景的环境由一个 Kubernetes 集群和若干虚拟机（VM）组成。如本节下文所述，ITBench 中预定义的每个 CISO 场景都由一个可部署栈（deployable stack）管理，它是一个软件组件，负责在真实运行时环境中处理评测流程与环境。

该可部署栈管理各种任务，包括为每次评测场景运行所做的准备（"deploy\_environment"）、故障注入（"inject\_fault"）、智能体性能评估（"evaluate"）和环境清理（"delete\_environment"）。每个可部署栈都是为特定的 CISO 合规评估场景专门设计的，确保所需的工具、配置和策略就位。环境管理员定义这些场景的可部署栈，并配置所需的软件，这可能涉及搭建策略引擎、工具，或创建模拟违反特定合规要求的条件。

例如，某个可部署栈可能故意展现一处错误配置（misconfiguration），以模拟违反某一特定合规标准。在一个场景的执行过程中，智能体最初被提供该场景合规要求的自然语言描述。基于该描述，智能体生成必要的产物，包括用于证据评估的脚本和策略。随后，运行时场景环境被部署并向智能体开放访问。智能体访问环境以检索证据、部署策略，并致力于实现指定的自动化目标——本例中即评估合规态势为"fail"或"not-satisfied"。

在场景运行过程中，智能体会向 ITBench 通报其任务的开始与完成。收到完成通知后，ITBench 访问环境以度量评测指标。一旦收集到所有预定义场景的指标，它们便被聚合并展示在 ITBench 排行榜（Leaderboard）上。图~\ref{fig:itbench-22} 展示了 CISO 场景的端到端评测流程。

在我们 CISO 合规评估评测的背景下，场景环境（包括 Kubernetes 集群和 VM）由智能体提交器（Agent Submitter）准备。搭建与指标度量使用一个名为 Mini-Bench 的工具自动化完成。评测结果（包括任务指标）通过 API 注册到中央 Bench Server。这些结果展示在 Bench Server 上的排行榜（Leaderboard）上。这一设置允许评测统一地进行——无论使用智能体提交器的本地环境还是远程环境，两种情况下采用的都是相同的 API 交互。

\subsection{D.5 CISO 场景类及其复杂度}

\subsubsection{D.5.1 New-K8s-CIS-b-Kyverno}

New-K8s-CIS-b-Kyverno 代表 ITBench 中的简单（Easy）场景类。该类中的 10 个场景基于面向 Kubernetes 的 CIS 基准（CIS Benchmark for Kubernetes）准备，具体聚焦于 Pod 安全策略（Pod Security Policy）。该场景假设一个预配置了 Kyverno 策略引擎的 Kubernetes 集群。在该集群内，存在某些与 Pod 安全策略相关的错误配置，但智能体并不知晓其确切位置。

需要处理的错误配置要求被传达给智能体。基于这些要求，智能体生成一个 Kyverno 策略并将其部署到集群。随后，智能体从集群收集报告。该报告的准确性通过检查它是否成功识别出错误配置来加以校验。如果策略被正确生成并部署，报告应指示出恰当的态势；反之，若出现错误，则会报告不正确的态势。

在该场景中，四项合规评估任务（Compliance Assessment Task）的评估方式如下：

\begin{itemize}
\item IEC：通过校验 Kyverno 策略中是否反映出正确的配置来评估。
\item IPA：通过确认是否成功为 Kyverno 生成了策略来评估。
\item CE：通过校验是否在预定义位置生成了 Kyverno 报告来度量。
\item SAP：由 Kyverno 报告中所报告的态势是否与期望值匹配来判定。
\end{itemize}

\subsubsection{D.5.2 New-K8s-CIS-b-OPARego}

New-K8s-CIS-b-Kubectl-OPARego 被归类为中等（Medium）复杂度场景。该基准包含 10 个源自面向 Kubernetes 的 CIS 基准的场景，具体聚焦于 Pod 安全策略。

该场景的基础背景与 New-K8s-CIS-b-Kyverno 高度相似，共享将 Kubernetes 集群作为运行环境、并将面向 Kubernetes 的 CIS 基准作为运行环境的假设。关键差异在于 Open Policy Agent（OPA）策略引擎的双重输出：既生成证据获取脚本（evidence fetcher script），也生成策略检查器代码（policy checker code）。

\begin{itemize}
\item 获取脚本（fetcher script）旨在通过执行 kubectl 命令从目标集群收集所需证据。
\item 策略检查器（policy checker）基于预定义规则校验所收集证据的合规性。这通过 Open Policy Agent（OPA）实现，并由 OPA Rego 策略定义。
\end{itemize}

该场景中智能体的目标是生成两个输出：1）一个执行 kubectl 命令的脚本（fetcher），2）一个用于合规校验的 OPA Rego 策略（checker）。

校验流程对智能体生成的输出评估如下：由 kubectl 命令组成的获取脚本针对一个真实的 Kubernetes 集群执行以收集证据。随后，所收集的证据使用生成的 OPA Rego 策略和 OPA 策略引擎加以评估，以校验结果是否与期望的合规结果一致。

该场景在以下四项合规评估任务上被评估：

\begin{itemize}
\item IEC：判定包含 kubectl 命令的获取脚本是否成功生成。
\item IPA：校验以 OPA Rego 策略实现的检查器是否被正确生成。
\item CE：检查针对集群执行获取脚本能否成功收集证据。
\item SAP：校验 OPA Rego 策略是否如期评估所收集证据，产出正确的合规评估。
\end{itemize}

这一方式为评估智能体生成有效脚本与策略以进行 Kubernetes 集群合规评估的能力提供了结构化评估。

\subsubsection{D.5.3 New-RHEL9-CIS-b-Ansible-OPA}

New-RHEL9-CIS-b-Ansible-OPA 属于中等（Medium）复杂度场景类，包含 20 个基于面向 RHEL9 操作系统的 CIS 基准的场景。该场景与 New-K8s-CIS-b-Kubectl-OPARego 共享一些共同特征，包括生成两份代码（一个获取脚本和一个检查器策略），以及使用 OPA Rego 策略进行合规校验。主要区别在于目标系统：与聚焦 Kubernetes 集群的 New-K8s-CIS-b-Kubectl-OPARego 不同，该场景以 RHEL9 主机为目标。因此，New-RHEL9-CIS-b-Ansible-OPA 不使用 kubectl 作为获取脚本，而是采用 Ansible playbook。该场景的目标是生成作为获取脚本的 Ansible playbook，以及作为检查器的 OPA Rego 策略。

校验流程对智能体生成的输出评估如下：获取脚本（Ansible playbook）针对一台真实的 RHEL9 主机执行以收集证据。随后，所收集的证据使用生成的 OPA Rego 策略在 OPA 策略引擎上分析，评估结果是否与期望的合规结果一致。

在该场景中，智能体的性能在以下四项合规评估任务上被评估：

\begin{itemize}
\item IEC：智能体是否以 Ansible playbook 的形式生成获取脚本？
\item IPA：智能体是否为检查器生成 OPA Rego 策略？
\item CE：Ansible playbook 能否针对 RHEL9 主机成功执行并收集相关证据？
\item SAP：生成的 OPA Rego 策略能否评估所收集证据并产出期望的合规结果？
\end{itemize}

该场景旨在评估智能体在 Kubernetes 之外的主机管理环境（具体聚焦于 RHEL9 系统）中执行合规评估任务的性能。

\subsubsection{D.5.4 Update-K8s-CIS-b-Kyverno}

Update-K8s-CIS-b-Kyverno 属于复杂度被归类为困难（Hard）的场景类，目前包含 10 个场景。与基于目标中指定的新要求生成新 Kyverno 策略的 New-K8s-CIS-b-Kyverno 不同，该场景涉及不同的目标。具体而言，它以一个既有的 Kyverno 策略作为输入，连同详细说明对原始要求所做修改的指令，生成一个满足修订后要求的\textit{已更新（updated）}策略。更新后的策略随后作为最终输出被部署（或更新）。

\subsubsection{D.6.1 评估指标}

我们的 CISO 智能体的效能，是基于其在场景类（随机注入了各种错误配置）的多样化环境中，检测产物错误配置（即不合规，例如未达到容器共享命名空间的最小数量，或数量超过阈值）或确认正确配置（即合规）的能力来度量的。

我们使用以下指标评估智能体检测（不）合规的有效性：

\begin{itemize}
\item 成功率使用 pass@1 量化。
\item 效率通过处理用时（Time to Process，TTP）刻画。
\end{itemize}

\subsubsection{D.6.2 指标定义}

\textbf{pass@1。}我们使用 pass@1 指标（Chen et al., 2021）评估智能体对态势"pass"/"fail"的正确评估，其定义如下：
\[
\mathrm{pass@}k := \mathbb{E}_{\text{Scenarios}}\left[1 - \frac{\binom{n-c}{k}}{\binom{n}{k}}\right]
\]
它是跨所有场景在 $k=1$ 次试验中正确性的无偏估计量。

\textbf{处理用时（Time to Process）。}我们识别出智能体成功判定态势的场景（即错误配置导致"fail"态势，而合规配置导致"pass"态势）。对于这些任务场景，我们计算 \textit{TTP}，用以度量智能体执行评估的耗时。否则，\textit{TTP} 被设为无穷大。

\subsubsection{D.6.3 评估结果}

\textbf{智能体总体结果。}总体而言，我们在表~5 和图~25 中的结果表明，基于 GPT 的模型在 pass@1 和处理用时两项指标上均占据主导地位。其 pass@1 几乎是次优模型（在 llama-3.1-405b-instruct 与 mistral-large-2 之间交替）的 2 倍，而 TTP 显示其能跨我们的场景类以最短时间处理这些场景。

\textbf{场景复杂度的影响。}CISO 场景的复杂度被直接映射到场景类。例如，Kyverno 场景为简单复杂度，k8s-opa 和 rhel-opa 为中等复杂度，而 Kyverno-update 场景为困难复杂度。不出所料，所有模型都表现挣扎——正如预期，随着场景难度从简单的 \textit{kyverno} 类升至困难的 \textit{kyverno-upadate} 类，难度逐渐增大。

\section{附录 E：FinOps}

\subsection{E.1 背景}

FinOps（财务运营，Financial Operations，即 Finance + Operations）是一种运营框架与文化实践，旨在最大化云的业务价值，并构建共担式的财务问责机制。其首要目标之一，是通过促进工程、财务与业务团队之间的协作，实现及时、数据驱动的决策。FinOps 由三个迭代阶段构成——告知（Inform，可见性与分配，Visibility \& Allocation）、优化（Optimize，费率与用量，Rates \& Usage）和运营（Operate，持续改进与用量，Continuous Improvement \& Usage）。重要的是，FinOps 的成功取决于：利用从应用性能管理（Application Performance Management，APM）、应用资源管理（Application Resource Management，ARM）以及财务（云成本管理，Cloud Cost Management）中收集到的实时洞察，进行多次迭代式调整（Storment and Fuller, 2023）。

与一般认知相反，FinOps 关注的是最大化业务价值，而不仅仅是降低运营成本。《2024 年 FinOps 现状报告》（Foundation, 2024a;b）从 1,245 名受访者处采集了数据，企业的平均年度云支出为 \$44M，部分企业的年度云支出甚至高达 \$1B 以上，云支出总额达 \$55B。该研究显示，与往年相比情况已发生转变：减少浪费、管理承诺（commitment）正成为各方新的关键优先事项。云折扣计划（如节省计划 Savings Plans 与预留实例 Reserved Instances）的管理，以及对支出的准确预测，仍位居榜单前列。诸如自动化程度提升、AI 成本、可持续性等其他领域也呈现出日益增长的关注度。

\subsubsection*{E.1.1 关键术语}

FinOps 通过在框架能力（Framework Capabilities）上迭代地开展工作，跨越三个阶段来实施（Foundation, 2025b），即告知（Inform）、优化（Optimize）与运营（Operate），分别描述如下：

\begin{enumerate}
\item 告知（Inform）阶段涉及识别云成本、用量与效率数据的数据源。随后，这些数据被用于预算编制、成本分配、预测、分析与报告。
\item 优化（Optimize）阶段利用在告知阶段开发出的数据与能力，识别提升云效率的机会。
\item 运营（Operate）阶段利用在告知与优化阶段开发出的数据与能力，将 FinOps 落地实施、运营化。
\end{enumerate}

FinOps 中一些适合用于优化的常见关键绩效指标（KPI）（完整集合见 \url{https://www.finops.org/wg/finops-kpis/}）包括：

\begin{enumerate}
\item 资源利用率百分比（Percentage Resource Utilization）——指已利用资源占总分配容量的百分比。
\item 支出的总未预测方差（Total Unpredicted Variance of Spend）——衡量在给定时间段内记录的、与云服务提供商（CSP）云用量相关的成本的未预测方差。
\item 自动扩缩效率比（Auto-scaling Efficiency Rate）——满足工作负载需求所需的最大运行容量成本 / 采用自动扩缩满足同等工作负载需求时的运行成本。
\item 预测准确率（用量）（Forecast Accuracy Rate (Usage)）——比较某一特定时段（如日、月、季度）内预测的云用量与实际云用量（vCPU、内存等）。
\item 预测准确率（支出）（Forecast Accuracy Rate (Spend)）——该指标比较某一特定时段（如日、月、季度）内预测的云支出与实际云支出。
\item 由承诺折扣覆盖的计算支出百分比（Percent of Compute Spend Covered by Commitment Discounts）——衡量某一特定时段内由承诺折扣覆盖的计算成本（不含 Spot）的百分比。
\item 承诺折扣浪费百分比（Percentage of Commitment Discount Waste）——未应用于按需支出的承诺所占的百分比。
\item 未使用资源百分比（Percent of Unused Resources）——衡量未使用的云资源，例如未挂载/孤立的存储卷、负载均衡器、EIPS、网络网关、快照。
\item 未分配的共享 CSP 云成本百分比（Percentage of Unallocated Shared CSP Cloud Cost）——该度量指无法直接归属到组织内特定项目、团队或部门的费用。
\item 已编制预算与已预测 CSP 云支出之间的方差百分比（Percentage Variance of Budgeted vs. Forecasted CSP Cloud Spend）——衡量使用 CSP 云服务的预算成本与预测成本之间的差异。
\item 有效节省率百分比（Effective Savings Rate Percentage）——含折扣的实际支出 / 等量的按需费率支出。
\item 符合标签策略的 CSP 云成本百分比（Percentage of CSP Cloud Costs that are Tagging Policy Compliant）——某一时段内符合标签策略的 CSP 云资源相关总成本 / 该时段内 CSP 云总成本。
\item 高频访问层上的存储百分比（Percent Storage on Frequent Access Tier）——标准（或“高频访问”）层中的 GB 数 vs.\ 存储的总 GB 数。
\item 与未打标签的 CSP 云资源相关的碳百分比（Percentage of Carbon Associated with Untagged CSP Cloud Resources）。
\end{enumerate}

还有若干与碳足迹（而非美元成本）相关的类似 KPI。在当前版本的评测基准中，我们的场景采用了一个基于支出方差的告警。其余 KPI 为构建大量额外场景提供了丰富的基础。

\subsection{E.3 用于构建 FinOps 任务场景的 ITBench 架构}

我们大量复用了为 SRE 场景所搭建的环境。我们采用 OpenCost 来监控成本，并在预定义的预算与效率阈值被突破时发出告警。OpenCost 是一款开源工具，它根据所分配的资源，将运行在 Kubernetes 部署所在节点上的虚拟机/节点（virtual machine/node）的成本进行分摊。它选取每个容器较高的利用率或请求数量，并采用一种负载分配策略（load distribution policy）来分摊成本。在我们的实验中，我们强制采用了一个自定义定价模型，主要包括单个 CPU 核心的每小时 CPU 成本费率，以及每 1 GB 内存的内存成本费率。其他成本组成部分（如网络成本与 Spot 实例定价）并非我们评测范围的重点。因此，我们将这些组件的成本视为可忽略不计。我们按照 SRE 场景中所述的方式，通过引入成本变化告警来模拟真实世界场景。

\subsection{E.4 评测}

我们沿用了 SRE-bench 中所使用的同一评测框架，并在场景中纳入了 OpenCost 与成本告警。

\subsubsection*{E.4.1 评测指标}

我们就两项主要任务对每个基于大语言模型（LLM）的智能体进行评测：（i）诊断（diagnosis）与（ii）缓解/修复（mitigation）。

\textbf{诊断（Diagnosis）。} 对智能体诊断能力的评测，依据其是否能够为预算告警提供准确的根因（root cause）。诊断包括智能体为识别预算告警根因所遵循的分析步骤。智能体在诊断方面的表现以 pass@1 来度量，该值表明提供给 SRE 的根因分析的准确性。

\textbf{缓解/修复（Mitigation）。} 缓解涉及推荐处置步骤，以清除告警并优化应用的成本与效率。我们使用基于 pass@1 分数的成功率来评测智能体，并使用邻近度分数（proximity scores）来分析智能体在实现 Section E.4.2 所定义的最优成本与效率指标方面的表现。诊断与缓解两类评测在未来都可以扩展，从而采用类似于 SRE-bench 评测的 LLM-as-a-judge（以大语言模型作为评判者）方式来自动度量智能体的表现。

\subsubsection*{E.4.2 指标定义}

\textbf{pass@1。} 对于智能体推荐的诊断与缓解步骤，我们采用 Section C.6.2 中所定义的同一指标。

\textbf{邻近度指标（Proximity metrics）。} 为分析智能体在实现最优成本与效率方面的表现，我们为每小时 CPU 成本、每小时内存成本、工作负载 CPU 效率以及工作负载内存效率定义了邻近度分数。邻近度分数通过计算所测量成本或效率指标的观测值与最优值之间的成比例绝对差，来刻画智能体的表现。我们将该成比例的值从 1 中减去，使得邻近度分数为 1 表示达到了最优表现。邻近度分数按如下方式计算：
\[
\mathrm{proximity}\_i = 1 - \frac{|\,\mathrm{observed}\_i - \mathrm{optimal}\,|}{\mathrm{optimal}}
\]
其中 $i$ 表示实验试验（experiment trial），$observed$ 是所测量指标的检索值，$optimal$ 是同一指标在基准真值（ground truth）中给定的值。

\textbf{每小时 CPU 成本（Hourly CPU cost）：} 分配单个 CPU 核心一小时的成本。我们通过 opencost 采用自定义定价策略，以确保基准真值不受不同云服务提供商之间或与本地部署（on-premise）之间定价变化的影响。

\textbf{每小时内存成本（Hourly Memory cost）：} 分配 1 GB 内存的成本。它同样采用自定义定价策略。

\textbf{工作负载 CPU 效率（Workload CPU efficiency）：} 我们取应用所有容器的平均 CPU 效率。我们通过将一个容器的 CPU 利用率除以其请求量（request amount）来计算该容器的 CPU 效率。例如，如果部署配置中的 CPU 请求为 100m，而容器实际使用 50m，则效率计算为 50。

\textbf{工作负载内存效率（Workload memory efficiency）：} 计算方式与 CPU 效率相同，即将利用率除以请求的内存量。
